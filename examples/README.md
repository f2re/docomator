# Учебные примеры Docomator

Все имена и сведения в этом каталоге вымышлены. Файлы не содержат рабочих персональных данных и не импортируются, не подключаются и не активируются автоматически.

## Быстрый сценарий

1. На экране «Сотрудники» откройте массовый импорт и выберите `data/employees.csv`.
2. Укажите «ФИО» как отображаемое имя, «Табельный номер» как колонку повторного сопоставления, а оставшиеся колонки создайте как понятные поля карточки.
3. В разделе «Шаблоны» подключите `templates/personal-card.docx`.
4. В строках карточки выделите только значения после подписей и сопоставьте четыре поля: ФИО, должность, подразделение и дату приёма. Подписи до выделения останутся неизменными.
5. Для сводного DOCX подключите `templates/team-register.docx`, сопоставьте четыре значения второй строки таблицы и включите повтор этой строки по участникам аудитории.
6. Для сводного XLSX подключите `templates/team-register.xlsx`, сопоставьте ячейки `A4:D4` и явно выберите эту использованную строку как повторяемый диапазон.
7. Сравните результат с файлами из `expected/`. Они показывают только ожидаемый вид и не являются эталоном побайтового результата.

## Состав

- `data/employees.csv` — три вымышленных сотрудника в UTF-8 и с заголовками на русском языке;
- `templates/personal-card.docx` — простой личный документ с четырьмя отдельными местами заполнения;
- `templates/team-register.docx` — сводная DOCX-таблица с одной безопасной строкой-образцом;
- `templates/team-register.xlsx` — сводная таблица с одной строкой-образцом без формул;
- `fixtures/header-field.docx` — минимальный DOCX с полем внутри верхнего колонтитула;
- `fixtures/scalar-fields.xlsx` — минимальный XLSX с четырьмя отдельными ячейками для скалярного заполнения;
- `fixtures/rejected/macro-part.docx` — инертный отрицательный пример с запрещённой частью макроса; рабочая проверка обязана отклонить его до сохранения;
- `expected/` — заполненные личный DOCX и сводные DOCX/XLSX для визуального сравнения.

## Происхождение и ограничения

Эти учебные OOXML-файлы детерминированно собраны из фиксированных локальных частей сценарием `scripts/ci/generate-examples.mjs`. Для положительных файлов проверка проекта требует точного SHA-256, закрытого списка частей, отсутствия макросов, формул, внешних связей, полей Word и внедрённых объектов. Затем рабочие парсеры выполняют проверку, структурный разбор, компиляцию, заполнение и обратное чтение выбранного текста DOCX, поля колонтитула, скалярного XLSX и повторяемых строк DOCX/XLSX. Отрицательный файл проверяется отдельно: ожидается точный отказ `macro_content`, и он не передаётся компилятору.

Отрицательный файл предназначен только для проверки отказа при загрузке. Не открывайте и не редактируйте его в офисном приложении: выберите его на шаге проверки документа и убедитесь, что сохранение недоступно.

Файлы не выдаются за документы, сохранённые конкретной версией LibreOffice или Microsoft Office. Совместимость с ними подтверждается только отдельной матрицей и фактическим прогоном на целевом Linux-стенде. До такого прогона это безопасный воспроизводимый учебный набор, а не закрытие P6.

Для осознанного обновления сначала измените локальные литералы генератора, затем выполните:

```bash
npm run build
npm run generate:examples
npm run check:examples
```

`manifest.sha256` создаётся вместе с файлами и проверяется также внутри автономного комплекта.
