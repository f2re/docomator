# Журнал изменений Docomator

## 2026-07-19 — P6: LibreOffice-gate на неизменяемом корпусе

Добавлено:

- LibreOffice-фаза больше не собирает упрощённые OOXML внутри gate: она читает пять фиксированных файлов из `examples/`;
- перед преобразованием каждого файла проверяются канонический путь, отсутствие символических ссылок, запись в `manifest.sha256` и точная SHA-256;
- Writer проверяется на личном DOCX, верхнем колонтитуле и repeat-DOCX, Calc — на scalar-XLSX и repeat-XLSX;
- отдельный тест production converter orchestration подтверждает полный состав из пяти файлов и очистку временных профилей без заявления о наличии настоящего LibreOffice.

P6 остаётся 🟡 до прогона этого же gate настоящим LibreOffice на целевых Debian/Astra, Office-сохранённого корпуса, актов восстановления и подготовки RC.

## 2026-07-19 — P4.3: HTTP-вход обязательного release-gate

Добавлено:

- загрузка CSV через настоящий HTTP endpoint, серверный preview, план без записи и keyless-импорт 10 сотрудников с последующим чтением записей из API;
- отдельные HTTP inspection, structure и quarantine для личного DOCX, repeat-DOCX и repeat-XLSX вместо прямой пометки исходников как принятых;
- проверка точного каталога трёх неизменяемых исходников до компиляции;
- сохранение прежнего production compiler/SQLite/object store, crash-worker, восстановления аренды, ZIP, repeat-DOCX/XLSX и reverse-read в том же обязательном gate.

PDF остаётся отдельной фазой LibreOffice: локальное отсутствие converter по-прежнему даёт явный `SKIPPED`, а target gate preview-комплекта запрещает такой пропуск.

## 2026-07-19 — P4.2: минимальный OOXML fixture-корпус

Добавлено:

- личный DOCX с четырьмя полями внутри строк после неизменяемых русских подписей; проверка использует `docx.text-range`, production-компиляцию, заполнение и обратное чтение;
- отдельный DOCX с полем верхнего колонтитула и проверкой сохранения основного документа;
- отдельный XLSX с четырьмя скалярными ячейками, техническим `_AI_META` после компиляции и обратным чтением каждого значения;
- инертный отрицательный DOCX с запрещённой частью макроса и обязательным точным отказом `macro_content` до сохранения;
- SHA-256 inventory из 10 файлов, точные allowlist автономной сборки/проверки и обновлённая русская инструкция.

P6 остаётся 🟡: корпус воспроизводимо создан Docomator и не заменяет очищенные Office-сохранённые документы, фактические LibreOffice/MS Office проверки и целевые акты Debian/Astra.

## 2026-07-19 — OFF-010/OFF-013: самодостаточный target bundle

Добавлено:

- обязательный явный профиль `--with-preview` или `--without-preview`; preview-сборка без проверенного набора LibreOffice Writer/Calc больше не создаётся;
- точные `manifest.sha256`, `packages.tsv` и `source-os.env` для `.deb` с повторной сверкой checksum, имени, версии и архитектуры через `dpkg-deb`;
- точное совпадение ID/версии/архитектуры target до package mutation, package-manager preflight замкнутости и запрет удаления либо сетевой загрузки пакетов;
- профиль, путь converter, пределы и SHA package inventory в `release.json` с согласованием с шаблоном конфигурации;
- root smoke, core/crash-worker/LibreOffice gates и единый `target-release-gate.sh` внутри production bundle без `npm ci`, registry и внешнего `curl`;
- ранний отказ installer, если preview включён, а настроенный LibreOffice недоступен; установленный помощник теперь получает локальный healthcheck;
- запрет совмещать обновление приложения с нетранзакционным обновлением пакетов ОС и защита `VERSION` от выхода за immutable releases root;
- негативные проверки отсутствующего Calc и обязательного gate, а также положительные проверки обоих preview-профилей.

Миграции и формат данных не менялись. Существующая конфигурация сохраняется; откат приложения выполняется предыдущим неизменяемым release-каталогом. P6 остаётся 🟡 до фактических актов чистых Debian/Astra/LibreOffice, Office-сохранённого корпуса, проверки восстановления, RC-версии и release notes.

## 2026-07-19 — AUD-003: единое скачивание результатов

Добавлено:

- ответы задания формирования возвращают идентификатор, карточку и ссылку скачивания канонического общего результата без отдельного состояния мастера;
- прежняя ссылка комплекта перенаправляет на единый маршрут, а скачивание отдельного файла использует тот же переход в состояние «забран»;
- каждая успешно подготовленная сервером выдача файла журналируется с correlation ID, видом файла и исходным состоянием, при этом итоговое доменное событие остаётся идемпотентным;
- отсутствие или повреждение объекта не переводит результат в «забран» и не создаёт ложную запись аудита;
- после успешного или частично успешного ручного выпуска интерфейс автоматически открывает «Результаты» и созданную карточку, откуда файл скачивается по единой ссылке;
- устаревший ответ предыдущего polling не может остановить новое задание, а переход во время занятого списка ставит обязательное обновление результатов в очередь;
- API- и Playwright-проверки фиксируют совместимую старую ссылку, состояние, аудит, отсутствие ложного успеха и поведение на ширинах 320, 768 и 1440 пикселей.

## 2026-07-19 — SEC-010: fail-closed проверка резервных копий

Добавлено:

- точное совпадение фактического состава backup с `manifest.sha256`, уникальность и каноничность каждого пути;
- отказ до восстановления при пропущенном или лишнем файле, повторе checksum-записи, symbolic link и специальном filesystem entry;
- строгая схема `manifest.json` и сверка размера БД, количества объектов и наличия конфигурации;
- установка данных только из локальной staging-копии после её повторной полной проверки;
- отделение успешной фиксации от best-effort удаления staging/rollback/lock-артефактов: ошибка очистки больше не может удалить уже установленную БД и возвращается как явное предупреждение;
- сохранение и возврат SQLite `-wal`/`-shm` при внутреннем откате;
- отрицательные тесты сценария с исключённым из manifest и изменённым object-файлом;
- проверка отката БД и object storage при ошибке восстановления конфигурации и удаления временных файлов.

## 2026-07-19 — P5.1: проверяемые автоматические свидетельства UX

Добавлено:

- полный Chromium E2E формирует Playwright JSON и отдельный агрегированный axe JSON;
- axe-отчёт сохраняет 12 обязательных проверок: шесть состояний для 320/light и 1440/dark с точным набором WCAG-тегов;
- `acceptance:ux collect-automation` fail-closed проверяет точный версионированный inventory из 81 Playwright-выполнения и 12 axe-состояний, сохраняет content-addressed копии и SHA-256 рядом с актом;
- каждый обычный `validate` повторяет семантическую проверку отчётов, поэтому корректный SHA-256 произвольного JSON не считается доказательством;
- axe `incomplete` превращается в обязательные ручные разборы по экрану и правилу; решение привязано к SHA-256 точного отчёта и не переиспользуется после изменения его байтов;
- каталог и исходный акт проверяются по владельцу/правам, а утверждённый акт не принимается для повторного сбора;
- сбор создаёт без перезаписи отдельный производный акт, изменяет в нём только `automationEvidence`, идемпотентен для одинаковых байтов и не переводит ручную приёмку в успешное состояние;
- прежние акты версии 1 без массива axe-разборов принимаются как legacy-вход и обновляют форму только в новом производном акте;
- повреждённые, неполные, упавшие отчёты и axe-нарушения отклоняются до создания производного акта.

P5 остаётся 🟡: фактические Linux-эталоны, экранный диктор, полные ручные маршруты и две независимые пользовательские сессии не выполнены.
