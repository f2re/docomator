# Финализация первого выпуска Docomator

Этот документ описывает последний fail-closed этап между готовым кодом и публикацией `0.1.0-rc.1`. Версия не повышается по факту прохождения CI: требуются фактические доказательства двух целевых ОС, ручной пользовательской приёмки, восстановления и совместимости реальных Office-документов.

## 1. Создать каркас доказательств

Из чистого checkout проверяемого commit:

```bash
npm run release:evidence:init -- /srv/docomator-release-evidence
```

Будет создана структура:

```text
/srv/docomator-release-evidence/
├── targets/
│   ├── debian/
│   └── astra/
├── ux/
│   └── ux-acceptance.json
├── recovery/
│   └── restore-act.json
├── office/
│   └── compatibility.json
├── blockers.json
└── README.md
```

## 2. Получить целевые акты

На чистых Debian и 🟥 Astra Linux выполните `target-acceptance.sh` из соответствующего полного offline bundle. Полный каталог результата копируется без изменения в `targets/debian` или `targets/astra`; внутри обязаны остаться `target-acceptance.json`, `manifest.sha256`, pilot JSON/Markdown, Playwright/axe и журналы всех этапов.

🟥 Astra Linux-прогон обязан использовать:

```bash
--require-network --require-smtp
```

## 3. Завершить ручную P5-приёмку

Заполните `ux/ux-acceptance.json` по `UX_ACCEPTANCE_PROTOCOL.md`. Акт должен содержать:

- Linux-среду и release binding одного из целевых прогонов;
- ручные проверки клавиатуры, фокуса, экранного диктора, масштаба и форм;
- шесть утверждённых PNG для 320/768/1440 × light/dark;
- Playwright и axe JSON того же target-прогона;
- два новых пользователя без обучения и помощи;
- итоговое решение `passed`.

Все файлы-свидетельства хранятся внутри каталога `ux` и проверяются по SHA-256 без символических ссылок.

## 4. Подтвердить восстановление

На отдельном чистом стенде восстановите копию, созданную целевой приёмкой. В `recovery/restore-act.json` укажите точный release, commit, SHA-256 исходной копии, источник Debian/Astra, фактические и ожидаемые количества участников, шаблонов, результатов и объектов, а также результат сравнения контрольных сумм.

## 5. Проверить реальные Office-документы

`office/compatibility.json` должен содержать не менее 20 уникальных DOCX и 20 уникальных XLSX. Для каждого файла фиксируются происхождение, программа-создатель, SHA-256 и успешное открытие результата в согласованных LibreOffice и Microsoft Office без видимых технических маркеров.

## 6. Закрыть блокирующие дефекты

`blockers.json` принимается только при пустом `openBlockers`. Не скрывайте дефекты сменой уровня: потеря данных, дубликаты, неправильный документ, неработающий update/rollback/restore, установка или основной путь остаются блокирующими.

## 7. Выполнить финальный gate

```bash
npm run release:evidence -- \
  /srv/docomator-release-evidence \
  --expected-commit '<полный Git SHA>' \
  --expected-version '0.1.0-alpha.0'
```

Gate принимает только точный состав и связи доказательств. Он блокирует выпуск при отсутствии любого файла, несовпадении SHA-256, версии или commit, старой резервной копии, неполной P5-приёмке, недостаточном Office-корпусе, расхождении после восстановления или открытом блокирующем дефекте.

## 8. Выпустить release candidate

Только после успешного gate:

1. изменить версию на `0.1.0-rc.1` во всех канонических местах;
2. выполнить полный `npm run check`;
3. собрать новые отдельные Debian и 🟥 Astra Linux bundles уже из RC commit;
4. повторить целевые акты для RC commit;
5. создать подписанный tag и опубликовать архивы, SHA-256, release notes и заполненную матрицу совместимости.

Обычная версия `0.1.0` выпускается после периода внутренней эксплуатации RC без блокирующих дефектов и повторно подтверждённого восстановления.
