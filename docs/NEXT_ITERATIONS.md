# Ближайшие приращения Docomator

Статус: **проверка приращения «поле и проверяемая привязка»**  
Последнее обновление: **2026-07-12**

План подчиняется [нормативным требованиям](REQUIREMENTS.md), [архитектуре](ARCHITECTURE.md) и общему [плану реализации](IMPLEMENTATION_PLAN.md). Каждое приращение должно оставлять систему запускаемой, автономной и пригодной для проверки без локальной модели.

## Ограничения, действующие для всех шагов

- содержимое DOCX/XLSX считается недоверенным;
- исходный XML, пути и команды из документа не исполняются;
- локальная модель не получает права изменять файлы, БД или внешние системы;
- данные и черновики не смешиваются между пространствами;
- каждое длительное действие имеет понятное русское состояние и следующий шаг;
- файл и активированная версия шаблона неизменяемы;
- новые действия с побочным эффектом получают идентификатор операции и защиту от повторов;
- интерфейс работает без внешних сетевых ресурсов и не требует знания английских терминов;
- новый шаг не считается завершённым без сборки, тестов, документации и проверки автономной поставки.

## Состояние ближайшей цепочки

| Шаг | Состояние | Проверяемый результат |
|---|---:|---|
| 1. Безопасная проверка пакета | ✅ | повреждённый, макросный или опасный файл отклоняется до сохранения |
| 2. Неизменяемый карантин | ✅ | принятый исходник сохраняется по SHA-256 внутри одного пространства |
| 3. Структурное представление | ✅ | абзацы DOCX и ячейки XLSX имеют устойчивые координаты и ограниченный ответ |
| 4. Сохранение поля | 🟡 | код, миграция, маршруты, интерфейс и тесты добавлены; выполняется полная проверка |
| 5. Скалярная компиляция | ⬜ | DOCX получает элемент управления содержимым, XLSX — именованную привязку |
| 6. Пробное заполнение | ⬜ | одно значение вставляется, затем считывается обратно и проверяется |
| 7. Версия шаблона | ⬜ | проверенный результат сохраняется как неизменяемая версия черновика |
| 8. Повторяющийся блок | ⬜ | `audience.members` выводится таблицей или списком в одном общем документе |
| 9. Документные задания | ⬜ | снимок аудитории создаёт один сводный либо несколько индивидуальных документов |

## Приращение A — структурное представление

**Состояние:** ✅ завершено и слито.

Реализовано:

1. DOCX: основной текст, верхние и нижние колонтитулы, сноски, абзацы, текстовые фрагменты и положение внутри таблицы;
2. XLSX: книга, листы, ячейки, общие строки, встроенные строки, значения и формулы;
3. идентификаторы, рассчитанные из SHA-256 исходника и структурной координаты;
4. запрет `DOCTYPE` и `ENTITY` до разбора XML;
5. ограничение числа элементов ответа от 10 до 2000;
6. явный признак сокращённого показа;
7. русскоязычный экран выбора элемента;
8. отрицательные тесты повреждённых частей и небезопасных объявлений XML.

Критерий завершения выполнен: одинаковый неизменяемый файл при повторном анализе возвращает те же координаты; крупный документ не перегружает браузер; исходный XML не передаётся пользовательскому интерфейсу.

## Приращение B — поле и проверяемая привязка

**Состояние:** 🟡 реализация добавлена, полная проверка выполняется.

Реализовано в текущем составе:

1. таблицы `template_drafts` и `template_draft_fields`;
2. принадлежность черновика одному пространству;
3. название, стабильный ключ, тип и обязательность поля;
4. сохранение `source_sha256`, `structure_sha256`, идентификатора элемента, координаты и исходного фрагмента;
5. запрет произвольной координаты, отсутствующего элемента и повторной скалярной привязки;
6. журнал действий и события `template.draft.created` / `template.draft.field.created`;
7. чтение исходника только через адресуемое по содержимому хранилище с повторной проверкой SHA-256;
8. маршруты создания/чтения черновика и сохранения поля;
9. интерфейс выбора типа, ключа, обязательности и подтверждения сохранения;
10. модульные и интеграционные тесты.

До отметки ✅ требуется:

- успешная строгая сборка TypeScript;
- успешные тесты хранилища и прикладного интерфейса;
- применение миграции `0006_template_drafts.sql` на чистой БД;
- проверка браузерного сценария и русской терминологии;
- проверка автономного комплекта и помощника первого запуска;
- обновление README, плана развития и основного ТЗ по фактическому результату.

Критерий завершения: пользователь выбирает элемент из построенной структуры, сохраняет поле, повторно открывает черновик и видит ту же проверенную привязку.

## Приращение C — базовый компилятор

**Цель:** превратить ручную привязку в техническую привязку Office Open XML без перестройки всего документа.

Входит:

1. DOCX: оборачивание целого абзаца или допустимого текстового фрагмента в `w:sdt` с технической меткой;
2. XLSX: создание именованной привязки к ячейке и служебных сведений;
3. проверка `source_sha256` и `structure_sha256` перед изменением;
4. сохранение нетронутых частей пакета;
5. повторное открытие результата и поиск созданной привязки;
6. неизменяемая скомпилированная копия с собственной SHA-256.

Критерий завершения: компилятор не изменяет исходник, воспроизводимо создаёт одну скалярную привязку и обнаруживает её при обратной проверке.

## Приращение D — пробное заполнение и активация

**Цель:** доказать, что поле не только размечено, но и заполняется без повреждения файла.

Входит:

1. ввод пробного значения по типу поля;
2. форматирование даты, числа, логического значения и строки;
3. вставка только в разрешённую привязку;
4. обратное чтение ожидаемого значения;
5. создание PDF-предпросмотра LibreOffice в отдельном ограниченном процессе;
6. отчёт проверки и запрет активации при ошибке;
7. неизменяемая версия шаблона.

Критерий завершения: неизвестный DOCX/XLSX проходит путь «проверка → разметка → компиляция → пробное значение → обратная проверка → активация» без участия модели.

## Приращение E — общий документ по аудитории

**Цель:** поддержать не только документ на каждого человека, но и один сводный документ.

Входит:

1. привязка повторяющейся строки, абзаца или диапазона;
2. источник коллекции `audience.members`;
3. устойчивый порядок участников из неизменяемого снимка;
4. таблица DOCX, список DOCX и повторяющаяся строка XLSX;
5. пустая аудитория как блокирующая ошибка;
6. предельное число строк и понятный отчёт;
7. тесты общего документа и режима «по документу на каждого» на одном наборе участников.

Критерий завершения: одна и та же аудитория воспроизводимо создаёт либо `N` индивидуальных файлов, либо один файл с `N` строками/элементами.

## После детерминированного пути

Только после выполнения A—E подключаются локальные агенты:

- поиск вероятных полей;
- сопоставление с существующими свойствами;
- предложение формата ФИО и даты;
- группировка уточняющих вопросов;
- создание разрешённых текстовых блоков.

Любое предложение агента остаётся данными, проходит схему и подтверждение, а файловые изменения выполняет только компилятор.
