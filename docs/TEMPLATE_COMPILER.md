# 🧱 Базовый компилятор шаблонов

Статус: **детерминированное ядро скалярной компиляции реализовано и проверено**  
Последнее обновление: **2026-07-12**

## Назначение

Компилятор превращает проверяемое поле черновика в техническую привязку Office Open XML. Он не заполняет документ данными и не активирует версию шаблона. Его задача — воспроизводимо внести одну разрешённую метку и доказать, что она присутствует после повторной сборки пакета.

```text
неизменяемый исходник
+ SHA-256 исходника
+ SHA-256 структуры
+ проверяемое поле
        ↓
повторный структурный анализ
        ↓
проверка координаты
        ↓
создание технической привязки
        ↓
повторная сборка ZIP
        ↓
обратное обнаружение привязки
```

Рабочая область:

```text
packages/template-compiler/
```

## Поддерживаемый базовый объём

### DOCX

Поддерживается поле, связанное с целым абзацем.

Компилятор оборачивает исходный абзац в элемент управления содержимым:

```xml
<w:sdt>
  <w:sdtPr>
    <w:alias w:val="ФИО получателя"/>
    <w:tag w:val="aifield:field-id"/>
    <w:id w:val="..."/>
  </w:sdtPr>
  <w:sdtContent>
    <w:p>...</w:p>
  </w:sdtContent>
</w:sdt>
```

Исходное содержимое абзаца сохраняется. `w:id` воспроизводимо рассчитывается из идентификатора поля.

### XLSX

Поддерживается поле, связанное с одной ячейкой.

В `xl/workbook.xml` создаётся именованная привязка:

```xml
<definedNames>
  <definedName name="_DOCOMATOR_...">'Сотрудники'!$B$7</definedName>
</definedNames>
```

Название листа экранируется по правилам формул Excel, адрес преобразуется в абсолютную ссылку. Содержимое и формула исходной ячейки не меняются.

## Входной контракт

```ts
interface CompileScalarFieldInput {
  source: Uint8Array;
  fileName: string;
  expectedSourceSha256: string;
  expectedStructureSha256: string;
  field: {
    id: string;
    key: string;
    label: string;
    elementId: string;
    binding: unknown;
  };
}
```

Перед изменением проверяются:

1. формат обеих контрольных сумм;
2. фактическая SHA-256 исходника;
3. повторно построенная SHA-256 структуры;
4. существование `elementId`;
5. совпадение вида и координаты элемента;
6. отсутствие уже созданной технической метки.

Браузер или локальная модель не могут передать произвольный путь части или координату без проверки по фактической структуре.

## Результат

```ts
interface CompileScalarFieldResult {
  output: Buffer;
  format: "docx" | "xlsx";
  sourceSha256: string;
  structureSha256: string;
  outputSha256: string;
  fieldId: string;
  fieldKey: string;
  modifiedPart: string;
  technicalBinding: {
    kind: "docx.sdt" | "xlsx.defined-name";
    identifier: string;
    part: string;
    target: string;
  };
  verification: {
    found: true;
    message: string;
  };
}
```

Исходный буфер не изменяется. Результат является новой копией с собственной SHA-256.

## Ограниченное чтение OOXML

Компилятор использует отдельный ограниченный модуль чтения ZIP:

- не доверяет только центральному каталогу;
- проверяет фактический размер каждой части;
- проверяет CRC-32 каждой части;
- ограничивает число частей, размер части и общий распакованный объём;
- запрещает зашифрованные части;
- разрешает только способы хранения и Deflate;
- запрещает абсолютные пути, `..`, обратные косые черты и управляющие символы;
- обнаруживает повторяющиеся имена.

## Воспроизводимая сборка

Повторная сборка:

- сохраняет порядок частей;
- использует фиксированные значения времени ZIP;
- использует UTF-8 для имён;
- повторно рассчитывает CRC-32;
- применяет воспроизводимый уровень Deflate;
- не переносит произвольные небезопасные атрибуты исходного архива.

Одинаковые исходник, поле и привязка дают одинаковые байты результата.

## Обратная проверка

После сборки результат открывается заново.

Для DOCX ищется точное значение `w:tag`.  
Для XLSX ищется точное имя `definedName`.

Если метка не найдена, операция завершается ошибкой и результат не считается скомпилированным.

## Что не входит в текущий объём

Пока не поддерживаются:

- фрагмент внутри абзаца;
- несколько полей за один вызов;
- строки и диапазоны таблиц;
- повторяющиеся блоки;
- изображения;
- условия;
- вставка пробного значения;
- обратное чтение значения;
- сохранение скомпилированной копии в БД;
- активация версии;
- предварительный просмотр LibreOffice.

Эти возможности добавляются последовательными приращениями, а не скрытым универсальным изменением XML.

## Проверки

Эталонные испытания подтверждают:

- создание `w:sdt` и сохранение текста DOCX;
- создание именованной привязки XLSX;
- экранирование апострофа в названии листа;
- сохранение нетронутых частей пакета;
- одинаковые байты при повторной компиляции;
- обратное обнаружение метки;
- отказ при неверной SHA-256 структуры;
- отказ при вымышленной координате;
- повторное чтение детерминированно собранного пакета.

## Следующее приращение

Следующий вертикальный шаг соединяет ядро с черновиком:

1. получить черновик и поле только внутри одного пространства;
2. прочитать исходник по SHA-256;
3. вызвать компилятор;
4. сохранить результат как неизменяемый объект;
5. записать отчёт, событие и аудит;
6. показать пользователю, какая часть изменена;
7. затем добавить пробное значение и обратное чтение.

## Связанные требования

`TPL-005`, `TPL-006`, `TPL-008`—`TPL-011`, `DOC-005`, `DOC-006`, `AUD-002`, `SEC-002`, `SEC-014`.
