# 🧱 Компилятор и пробное заполнение шаблонов

Статус: **скалярная компиляция, безопасный диапазон DOCX, форматтеры, многополевая проверка, PDF-предпросмотр и активация реализованы**
Последнее обновление: **2026-07-16**

## Назначение

Детерминированное ядро превращает проверяемое поле черновика в техническую привязку Office Open XML, записывает пробное значение только в отдельную копию, считывает значение обратно и сохраняет проверенную версию.

```text
неизменяемый исходник
+ SHA-256 исходника
+ SHA-256 структуры
+ проверяемое поле
        ↓
повторный структурный анализ
        ↓
техническая привязка
        ↓
пробное значение
        ↓
обратное чтение
        ↓
точное сравнение
        ↓
две неизменяемые копии
```

Рабочая область:

```text
packages/template-compiler/
```

## Поддерживаемый объём

### DOCX

Поддерживается совместимое поле, связанное с целым абзацем, и основное поле, связанное с выбранным текстом внутри одного либо нескольких соседних одинаково оформленных `w:r`.

Компилятор оборачивает исходный абзац в элемент управления содержимым:

```xml
<w:sdt>
  <w:sdtPr>
    <w:alias w:val="ФИО получателя"/>
    <w:tag w:val="aifield:field-id"/>
    <w:id w:val="..."/>
  </w:sdtPr>
  <w:sdtContent>
    <w:p>...</w:p>
  </w:sdtContent>
</w:sdt>
```

При пробном заполнении:

- сохраняются свойства абзаца `w:pPr`;
- сохраняются свойства первого текстового фрагмента `w:rPr`;
- старое отображаемое значение заменяется только внутри `w:sdtContent`;
- остальные части документа остаются нетронутыми;
- значение считывается обратно из технической привязки.

Для `docx.text-range` только выбранный фрагмент помещается в run-level `w:sdt`; подпись и текст до/после остаются снаружи. Компилятор отклоняет смешанное оформление, гиперссылки, поля, рисунки, разрывы и другие сложные объекты внутри диапазона. Границы хранятся как UTF-16 offsets вместе с точным исходным текстом и повторно проверяются по неизменяемой структуре.

### XLSX

Поддерживается поле, связанное с одной ячейкой.

В `xl/workbook.xml` создаётся именованная привязка:

```xml
<definedNames>
  <definedName name="_DOCOMATOR_...">'Сотрудники'!$B$7</definedName>
</definedNames>
```

При пробном заполнении:

- сохраняется стиль ячейки;
- строка записывается как встроенная строка;
- число — как числовое значение;
- логическое значение — как `0`/`1` с русским обратным отображением «Нет»/«Да»;
- запись поверх формулы запрещена;
- значение считывается обратно из той же ячейки.

## Типы пробного значения

Поддерживаются:

| Тип | Проверка и представление |
|---|---|
| Короткая строка | до 4000 знаков |
| Длинный текст | до 20 000 знаков |
| Число | конечное число; канонический ввод принимает точку или запятую, отображение использует русскую запятую и 0—6 настроенных знаков |
| Целое число | без дробной части |
| Да / нет | только логическое значение |
| Дата | канонический ввод `ГГГГ-ММ-ДД`, отображение `ДД.ММ.ГГГГ` |
| Дата и время | разбираемое каноническое значение, отображение `ДД.ММ.ГГГГ ЧЧ:ММ` в сохранённом IANA timezone |

Пустое или недопустимое пробное значение не создаёт проверенную версию.

## Ограниченный реестр форматтеров

Форматтер — версионированные JSON-данные, а не выражение или пользовательский код. Поддерживаются только:

| Вид | Параметры | Отображение |
|---|---|---|
| `identity` | нет | строка без преобразования |
| `number.ru` | `fractionDigits`: `null` или `0…6` | десятичная запятая, без группировки |
| `date.ru` | нет | `ДД.ММ.ГГГГ` |
| `date-time.ru` | проверенный IANA `timeZone` | `ДД.ММ.ГГГГ ЧЧ:ММ` |
| `boolean.ru` | нет | `Да` / `Нет` |
| `legacy` | только миграционный fallback | прежнее отображение уже созданных полей |

Backend выбирает формат по типу поля и отклоняет несовместимый вид, неизвестную версию, более шести десятичных знаков и неподдерживаемый часовой пояс. Контракт переносится без изменения из черновика в многополевую проверку, кандидата, активный манифест и production generation. Для необязательного отсутствующего значения worker записывает пустую строку без запуска несовместимого форматтера.

В XLSX число остаётся числовым `<v>` с канонической точкой; русское отображение используется для детерминированного обратного сравнения. Строковые календарные и логические значения записываются в уже отформатированном виде.

## Проверки перед компиляцией

1. формат SHA-256;
2. фактическая SHA-256 исходника;
3. повторно построенная SHA-256 структуры;
4. существование `elementId`;
5. совпадение вида и координаты элемента;
6. отсутствие ранее созданной технической метки;
7. соответствие технической привязки сохранённому виду поля;
8. отсутствие формулы в целевой ячейке XLSX.

Браузер и локальная модель не могут передать произвольный путь части или координату без проверки по фактической структуре.

## Воспроизводимая сборка OOXML

Модуль чтения и записи ZIP:

- проверяет фактический размер каждой части;
- проверяет CRC-32;
- ограничивает число частей и суммарный объём;
- запрещает шифрование и опасные пути;
- сохраняет порядок частей;
- использует фиксированные значения времени ZIP;
- повторно рассчитывает CRC-32;
- применяет воспроизводимое сжатие;
- не переносит произвольные небезопасные атрибуты архива.

Одинаковые исходник, поле, привязка и значение дают одинаковые байты соответствующей копии.

## Обратная проверка

После компиляции:

- DOCX повторно открывается и ищется точное значение `w:tag`;
- XLSX повторно открывается и ищется точное имя `definedName`.

После пробного заполнения:

- значение считывается из `w:sdtContent` или целевой ячейки;
- считанное отображение сравнивается с нормализованным ожидаемым значением;
- при расхождении операция завершается ошибкой;
- проверенная версия не сохраняется.

## Проверенная версия

Миграция:

```text
migrations/0007_template_test_versions.sql
```

Добавочная миграция `0023_template_field_formatters.sql` добавляет ограниченный `formatter_json` в черновики, многополевые строки и кандидатов. Существующие строки получают `legacy`, поэтому прежние активные версии сохраняют результат. Применённые миграции не изменяются; rollback выполняется восстановлением предварительной резервной копии базы вместе с предыдущей версией приложения.

Для успешной проверки сохраняются две разные неизменяемые копии:

1. **копия с технической привязкой** — исходное отображаемое значение ещё не заменено;
2. **пробно заполненная копия** — содержит проверенное значение.

Запись хранит:

- пространство;
- черновик и поле;
- последовательный номер версии;
- SHA-256 обеих копий;
- техническую привязку;
- пробное значение;
- записанное и считанное значения;
- отчёт проверки;
- автора, время и идентификатор операции.

Запись неизменяема. Повтор той же проверки с теми же байтами и значением не создаёт дубликат.

## Прикладной интерфейс

### Выполнить пробное заполнение

```http
POST /api/v1/spaces/{spaceId}/template-drafts/{draftId}/trial
Content-Type: application/json

{
  "fieldId": "...",
  "value": "Иванов Иван Иванович"
}
```

### История проверенных версий

```http
GET /api/v1/spaces/{spaceId}/template-drafts/{draftId}/test-versions
GET /api/v1/spaces/{spaceId}/template-test-versions/{versionId}
```

### Скачать копии

```http
GET /api/v1/spaces/{spaceId}/template-test-versions/{versionId}/files/compiled
GET /api/v1/spaces/{spaceId}/template-test-versions/{versionId}/files/trial
```

Скачивание предварительно проверяет принадлежность версии пространству и фактическую SHA-256 объекта.

## Пользовательский процесс

```text
поле сохранено
      ↓
выбрать черновик и поле
      ↓
ввести пробное значение
      ↓
«Создаём техническую привязку»
      ↓
«Записываем и считываем обратно»
      ↓
✅ точное совпадение
      ↓
скачать две копии
```

При ошибке введённое значение остаётся в форме, версия не сохраняется, пользователь видит причину и идентификатор операции.

## Предварительный просмотр и активная версия

Проверенная пробная версия передаётся в сохраняемое задание `template.preview`. LibreOffice запускается без оболочки и графического интерфейса, с отдельным временным профилем, ограничением времени, размера и захватываемого вывода.

PDF проверяется по сигнатуре и SHA-256. При ошибке проверенная пробная копия сохраняется, пользователь видит причину и может явно создать новую попытку.

После просмотра PDF пользователь отдельно подтверждает активацию. Активная версия неизменяема и содержит скомпилированный файл, PDF и манифест версии 3 с полным набором проверенных полей и форматтеров. Текущая версия определяется отдельным указателем каталога пространства.

Подробный контракт: [предварительный просмотр и активация](TEMPLATE_ACTIVATION.md).

## Ограничения текущего объёма

Пока не поддерживаются:

- несколько скалярных полей внутри одного абзаца;
- строка или диапазон таблицы;
- повторяющиеся блоки;
- изображения;
- условия;
- визуальное сравнение нескольких PDF;
- пользовательская повторяющаяся строка сводного документа по `audience.members`.

## Проверки

Эталонные испытания подтверждают:

- создание `w:sdt` и сохранение текста/оформления DOCX;
- создание именованной привязки XLSX;
- экранирование названия листа;
- сохранение стиля ячейки;
- запись и обратное чтение строки, числа и логического значения;
- русское отображение числа, даты, даты-времени и логики, включая явный часовой пояс;
- сохранение числового типа ячейки XLSX при форматированном обратном чтении;
- перенос неизменяемого форматтера в проверенную и активную версии;
- сохранение прежнего результата существующих строк после миграции;
- отказ от записи поверх формулы;
- проверку дат и целых чисел;
- сохранение нетронутых частей пакета;
- воспроизводимость результата;
- устранение повторов проверенной версии;
- неизменяемость записи в SQLite;
- изоляцию по пространствам;
- скачивание обеих копий через прикладной интерфейс.

## Следующее приращение

1. отдельный тип привязки повторяющегося блока;
2. одна строка таблицы DOCX из `audience.members`;
3. ограничение 1000 строк и структурная обратная проверка;
4. аналогичная непрерывная строка или диапазон XLSX.
