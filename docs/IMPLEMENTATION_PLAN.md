# План реализации Docomator

План организован по работающим вертикальным приращениям. Каждый этап должен оставлять систему запускаемой, проверяемой и пригодной для демонстрации. Ближайшая подробная последовательность находится в [NEXT_ITERATIONS.md](NEXT_ITERATIONS.md).

## Правила выполнения

- Каждое изменение ссылается на идентификаторы из [REQUIREMENTS.md](REQUIREMENTS.md).
- Сначала реализуется детерминированное ядро, затем ускорение локальной моделью.
- Содержимое DOCX/XLSX всегда считается недоверенным.
- Локальная модель не получает прямого доступа к файлам, БД, командам оболочки и внешним действиям.
- Изменения одного пространства не читают и не изменяют данные другого пространства.
- Длительные операции не удерживают транзакцию SQLite.
- Новая рабочая зависимость требует обоснования; архитектурное изменение — отдельного ADR.
- Этап не закрывается без тестов, документации, заметок о миграции/откате и оценки автономной поставки.
- Пользовательская функция проектируется вместе с состояниями загрузки, пустого списка, успеха, ошибки и ограниченной работы.
- Проверяются телефон, клавиатура, фокус и русский текст следующего шага.
- Необязательные англицизмы в интерфейсе, сообщениях и помощнике установки запрещены.

## M2. Безопасный приём, структура и поля — ✅ выполнено

**Требования:** TPL-001—004, TPL-013—023, SPACE-016, SEC-002, SEC-009, SEC-012—014, UX-027—028, AC-021—023.

### M2.1. Безопасность пакета

Реализовано:

- ограничения размера файла, числа частей и суммарного объёма;
- запрет опасных путей, шифрования и символических ссылок;
- поиск дубликатов и неоднозначных имён;
- обнаружение макросов, ActiveX, вложений, подписей и внешних связей;
- проверка обязательных частей DOCX/XLSX;
- русскоязычный отчёт «принят / с замечаниями / отклонён»;
- отрицательные испытания архивной структуры.

Отдельным усилением остаётся независимый общий предел фактически прочитанного потока всех частей перед долговременным хранением.

### M2.2. Неизменяемый карантин

Реализовано:

- отдельное подтверждение сохранения;
- повторная серверная проверка;
- хранение по SHA-256;
- изоляция по пространствам;
- устранение дубликатов;
- полный отчёт, автор, время и идентификатор операции;
- событие предметной области и аудит;
- список сохранённых исходников.

### M2.3. Структурное представление

Реализовано:

- DOCX: основной текст, колонтитулы, сноски, абзацы, текстовые фрагменты и координаты внутри таблицы;
- XLSX: книга, связи, листы, ячейки, общие/встроенные строки, значения, типы и формулы;
- устойчивые идентификаторы из SHA-256 исходника и структурной координаты;
- контрольная сумма структуры;
- запрет `DOCTYPE` и `ENTITY`;
- ограничение ответа 10—2000 элементов и явный признак сокращённого показа;
- прикладной интерфейс анализа и русскоязычный экран выбора элемента;
- модульные, интеграционные и отрицательные тесты.

### M2.4. Черновики и проверяемые поля

Реализовано:

1. миграция `0006_template_drafts.sql`;
2. таблицы черновиков и полей;
3. принадлежность черновика одному пространству;
4. создание черновика только из карантинного исходника;
5. чтение исходника через хранилище по SHA-256 с повторной проверкой;
6. серверный структурный анализ перед созданием черновика;
7. название, стабильный ключ, тип и обязательность поля;
8. сохранение `source_sha256`, `structure_sha256`, `element_id`, координаты и исходного фрагмента;
9. запрет произвольной координаты, отсутствующего элемента, повторного ключа и повторной скалярной привязки;
10. события `template.draft.created` и `template.draft.field.created`;
11. аудит и идентификатор операции;
12. маршруты создания/чтения черновика и сохранения поля;
13. интерфейс формы поля с русскими подсказками и состояниями;
14. модульные и интеграционные тесты;
15. проверка миграции, браузерного кода, терминологии, оболочечных сценариев и автономной поставки.

Критерий M2 выполнен: пользователь проверяет файл, сохраняет исходник, строит структуру, выбирает элемент и сохраняет типизированное поле; сервер повторно читает исходник и фиксирует собственную проверенную координату.

### M2.5. Сравнение примеров — перенесено в развитие студии

После базового компилятора будут добавлены:

- сравнение пустого шаблона с заполненным экземпляром;
- сравнение нескольких заполненных экземпляров;
- детерминированные кандидаты изменяемых областей;
- кандидат повторяющегося блока для `audience.members`.

## M3. Компилятор шаблонов и безопасное скалярное заполнение — ⬜ следующий этап

**Требования:** TPL-005—012, DOC-005—010, SPACE-013.

### M3.1. Базовая техническая привязка

1. определить контракт входа компилятора: исходник, `source_sha256`, `structure_sha256`, поле и привязка;
2. DOCX: обернуть целый абзац в `w:sdt` с технической меткой;
3. XLSX: создать именованную привязку к одной ячейке;
4. проверить контрольные суммы до изменения;
5. сохранить нетронутые части пакета;
6. повторно открыть результат и найти созданную привязку;
7. сохранить скомпилированную копию по SHA-256;
8. сформировать понятный отчёт компиляции.

Критерий: исходник не изменяется, одна скалярная привязка создаётся воспроизводимо и обнаруживается при обратной проверке.

### M3.2. Пробное заполнение

1. реестр форматтеров строки, числа, целого, логического значения, даты и даты-времени;
2. вставка только в разрешённую привязку;
3. обратное чтение ожидаемого значения;
4. структурная проверка ZIP/XML;
5. отчёт незаполненных или потерянных привязок.

### M3.3. Версия и предварительный просмотр

1. неизменяемая версия шаблона;
2. пробные данные;
3. LibreOffice без интерфейса с отдельным профилем и ограничением времени;
4. PDF-предпросмотр;
5. запрет активации при ошибке;
6. история версий и откат выбора активной версии.

### M3.4. Повторяющийся блок аудитории

1. повторяющийся абзац/строка таблицы DOCX;
2. повторяющаяся строка или диапазон XLSX;
3. источник `audience.members`;
4. устойчивый порядок участников из снимка;
5. пустая аудитория как блокирующая ошибка;
6. испытания одного сводного и нескольких индивидуальных документов.
