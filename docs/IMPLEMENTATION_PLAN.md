# План реализации Docomator

План организован по работающим вертикальным приращениям. Каждый этап должен оставлять систему запускаемой, проверяемой и пригодной для демонстрации.

## Правила выполнения

- Каждое изменение ссылается на идентификаторы из [REQUIREMENTS.md](REQUIREMENTS.md).
- Сначала реализуется детерминированное ядро, затем ускорение локальной моделью.
- Новая рабочая зависимость требует обоснования; архитектурное изменение — отдельного решения ADR.
- Этап не закрывается без тестов, документации, заметок о миграции/откате и оценки автономной поставки.
- Пользовательская функция проектируется вместе с состояниями загрузки, пустого списка, успеха, ошибки и ограниченной работы.
- Для каждой пользовательской функции проверяются телефон, клавиатура, фокус и русский текст следующего шага.
- Необязательные англицизмы в интерфейсе, сообщениях и установочном помощнике запрещены.
- Сохранение файла, внешняя доставка и иные побочные действия выполняются только после явного или заранее утверждённого решения.

## M2. Безопасный приём, карантин и структурное представление — 🟡 выполняется

**Требования:** TPL-001—004, TPL-014—020, SEC-002—003, SEC-009, SEC-012—013, UX-027, AUD-007, AC-021—022.

### M2.1. Первичная проверка — ✅ выполнено

1. отдельный модуль безопасного чтения ZIP без распаковки в пользовательские каталоги;
2. ограничения размера файла, числа частей, отдельной части и суммарного объёма;
3. запрет выхода пути, шифрования, символических ссылок, дубликатов и неоднозначных имён;
4. обнаружение макросов, ActiveX, вложенных объектов, подписей и внешних связей;
5. проверка обязательных частей DOCX/XLSX;
6. русскоязычный отчёт с состояниями «прошёл», «с замечаниями», «отклонён»;
7. прикладной интерфейс проверки двоичного файла;
8. сопровождающий экран выбора файла и просмотра результата;
9. модульные, отрицательные и интеграционные проверки;
10. включение рабочего модуля в автономную поставку.

### M2.2. Неизменяемый карантин — ✅ выполнено

1. отдельное подтверждение сохранения после первичной проверки;
2. повторная серверная проверка фактических байтов;
3. вычисление и проверка SHA-256;
4. хранение файла в адресуемом по содержимому хранилище;
5. неизменяемая запись карантина;
6. привязка записи к одному пространству;
7. устранение физического дубликата и дубликата записи в том же пространстве;
8. сохранение полного отчёта, автора, времени и идентификатора операции;
9. транзакционное событие и запись аудита;
10. API списка и получения исходников;
11. интерфейс выбора пространства, сохранения и истории исходников;
12. запрет помещения отклонённого файла в карантин.

### M2.3. Фактическая целостность частей — ⬜ следующий шаг

1. потоково читать содержимое каждой части;
2. подтверждать фактический распакованный размер;
3. проверять CRC всех частей;
4. ограничивать суммарный объём независимо от метаданных центрального каталога;
5. проверять локальный заголовок и метаданные центрального каталога на согласованность;
6. покрыть повреждённые потоки и расхождения размеров отрицательными тестами.

Критерий завершения: повреждение фактических данных обнаруживается до сохранения исходника, даже если центральный каталог выглядит допустимым.

### M2.4. Структурное представление DOCX — ⬜

1. части документа, верхние/нижние колонтитулы, сноски и концевые сноски;
2. абзацы, текстовые фрагменты и базовые свойства начертания;
3. таблицы, строки и ячейки;
4. устойчивые идентификаторы и структурные координаты;
5. ограничение количества элементов в ответе;
6. отсутствие исходного XML в пользовательском ответе;
7. эталонные документы с разделёнными фрагментами и вложенными таблицами.

### M2.5. Структурное представление XLSX — ⬜

1. книга, листы и отношения;
2. ячейки, типы значений, общие строки и формулы;
3. таблицы и именованные диапазоны;
4. устойчивые идентификаторы и адреса;
5. ограничение количества элементов в ответе;
6. эталонные книги с формулами, объединениями и повторяющимися строками.

### M2.6. Ручная разметка поля — ⬜

1. структурный просмотр с выбором абзаца, фрагмента, ячейки, строки или диапазона;
2. создание локального поля шаблона;
3. сопоставление с существующим свойством либо предложение нового;
4. сохранение проверяемой структурной координаты;
5. повторное открытие разметки без потери координат;
6. состояния пустого документа, недоступного элемента, конфликта и устаревшей структуры.

### M2.7. Сравнение примеров и повторяющиеся блоки — ⬜

1. сравнение пустого шаблона с одним заполненным экземпляром;
2. сравнение нескольких заполненных экземпляров;
3. детерминированные кандидаты изменяемых областей;
4. кандидат повторяющейся строки/таблицы;
5. подготовка блока для `audience.members`;
6. подтверждение кандидатов пользователем до сохранения.

Критерий завершения M2: пользователь проверяет файл, подтверждает неизменяемое хранение, открывает структурный просмотр и вручную сохраняет поле по проверяемой координате. Тот же процесс работает без локальной модели.
