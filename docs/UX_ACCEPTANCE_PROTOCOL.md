# Протокол ручной UX-приёмки

Статус: **инструментарий и безопасный сбор автоматических доказательств реализованы; фактическая ручная приёмка не выполнена**

Последнее обновление: **2026-07-19**

## Назначение

Автоматические Playwright и axe-проверки не доказывают доступность для экранного диктора и не заменяют действия новых пользователей. Этот протокол собирает обезличенный, проверяемый по SHA-256 акт P5, но не утверждает его автоматически.

Связанные требования: `UX-015`—`UX-022`, `UX-032`—`UX-036`, `NFR-011`, `NFR-012`, `AC-015`, `AC-016`, `AC-020`, `AC-027`—`AC-029`, `UX-AC-001`—`UX-AC-017`.

## Создание акта

На каноническом Ubuntu/Astra-стенде:

```bash
npm run acceptance:ux -- init ux-acceptance.json
```

Новый файл намеренно имеет статус `pending` и не может пройти проверку без ручных свидетельств. Повторный `init` не перезаписывает существующий акт.

В акте запрещены имена, адреса и иные персональные данные. Используются только безопасные псевдонимы вида `participant-alpha` и `reviewer-01`.

## Обязательная среда

- `platform` строго `linux`;
- версия Ubuntu/Astra и архитектура в `operatingSystem`;
- точная версия Chromium;
- название и версия экранного диктора, например Orca;
- полный SHA-1 (40 знаков) или SHA-256 (64 знака) проверяемого git commit;
- время проверки в каноническом UTC ISO-формате, не из будущего.

Локальный снимок macOS не считается утверждённым эталоном.

## Ручная матрица доступности

Каждый пункт получает `status: passed`, время и содержательное свидетельство:

1. полный клавиатурный маршрут;
2. порядок, trap и возврат фокуса;
3. экранный диктор и живые объявления;
4. системный масштаб текста 200%;
5. реальные зоны нажатия;
6. сохранение формы после ошибки;
7. показ идентификатора операции.

`failed` делает акт неуспешным; `pending` оставляет его незавершённым. Отчёт axe не может заменить эти ручные строки.

## Визуальные эталоны

Требуются шесть реальных PNG: 320, 768 и 1440 пикселей в светлой и тёмной темах. Для каждого указываются относительный путь внутри каталога акта, SHA-256, время утверждения и псевдоним проверяющего.

Валидатор проверяет:

- наличие ровно шести координат;
- нормализованный относительный путь без `..`, управляющих символов и символических ссылок в любом компоненте;
- обычный файл до 50 МБ;
- PNG-сигнатуру и точную ширину;
- SHA-256 фактических байтов.

Снимки из обычного диагностического режима Playwright не считаются эталонами до явного утверждения на Linux.

## Автоматические отчёты

Акт требует два JSON-свидетельства: полный Playwright-report и отдельный axe-report. Они хранятся рядом с актом, имеют относительные пути, временные отметки и SHA-256. Изменение файла после заполнения акта обнаруживается валидатором.

На запущенном локальном Docomator канонический браузерный прогон формирует оба файла без сетевых обращений за пределы настроенного origin:

```bash
DOCOMATOR_E2E_BASE_URL=http://127.0.0.1:18080 \
  npm run test:e2e
```

Результаты находятся в `tests/e2e/.tmp/playwright-report.json` и `tests/e2e/.tmp/axe-report.json`. Первый отчёт обязан подтверждать точный версионированный inventory: 27 сценариев в каждом из проектов 320, 768 и 1440 пикселей, всего 74 успешных и семь заранее зафиксированных неприменимых выполнений. Частичный запуск через `--grep`, новый необъявленный skip, нестабильный или лишний сценарий не принимаются. Второй отчёт содержит ровно 12 axe-проверок: шесть обязательных состояний на 320 пикселях в светлой теме и 1440 пикселях в тёмной, с полным набором WCAG-тегов и без нарушений.

После успешного прогона отчёты добавляются к уже созданному акту отдельной opt-in командой:

```bash
npm run acceptance:ux -- collect-automation \
  ux-acceptance.json \
  ux-acceptance-with-automation.json \
  tests/e2e/.tmp/playwright-report.json \
  tests/e2e/.tmp/axe-report.json
```

Первый путь после команды — исходный акт, второй — новое имя производного акта в том же каталоге. Исходный акт никогда не изменяется. Акты версии 1, созданные прежним `init` без массива `automationEvidence[].reviews`, читаются как legacy-форма с пустым списком разборов; актуальная форма появляется только в новом выходном акте. Сборщик отклоняет повреждённый, неполный или упавший отчёт до создания производного акта. Принятые файлы сохраняются рядом с актом под content-addressed именами, записываются без перезаписи, получают SHA-256 и проверяются существующим валидатором; каждый последующий `validate` заново проверяет не только байты, но и полный семантический контракт обоих отчётов.

Если axe не может автоматически решить применимость правила и возвращает `incomplete`, сборщик создаёт внутри `automationEvidence` отдельную обязательную ручную проверку для каждой пары «экран + правило». Проверка привязана к SHA-256 точного axe-отчёта; время ручного решения не может предшествовать формированию отчёта. При изменении байтов отчёта прежнее решение не переиспользуется и возвращается в `pending`. Проверяющий указывает `passed`, UTC-время, безопасный псевдоним и содержательное свидетельство; неизвестный или пропущенный unresolved-результат закрыть акт не позволяет. Команда не изменяет ручную матрицу, визуальные утверждения, участников и итоговое решение, поэтому незавершённый акт остаётся `incomplete`.

Каталог и исходный акт должны принадлежать пользователю, запускающему команду, и не разрешать запись группе или остальным. Производный акт создаётся без перезаписи существующего пути: повтор с теми же байтами идемпотентен, а существующий отличающийся файл блокирует сбор. Так параллельная ручная правка не может быть потеряна. Утверждённый исходный акт с `decision.status: passed` не принимается для повторного сбора.

## Два новых пользователя

Требуются два разных участника, которые:

- впервые используют Docomator;
- не являются участниками разработки;
- не проходили предварительное обучение;
- не получают устных подсказок во время заданий (`assistanceEvents: 0`).

Каждый участник ровно один раз выполняет три задания:

1. добавить сотрудника и поле;
2. подключить шаблон;
3. выпустить личные документы.

Для задания фиксируются начало, завершение и обезличенное свидетельство результата. Время завершения не может предшествовать началу.

## Проверка

```bash
npm run acceptance:ux -- validate ux-acceptance-with-automation.json
npm run acceptance:ux -- validate ux-acceptance-with-automation.json --json
```

Коды завершения:

| Код | Значение |
|---:|---|
| `0` | акт полный, все заявленные проверки и файлы целостны |
| `1` | акт корректен, но свидетельства ещё не заполнены |
| `2` | акт повреждён, содержит отказ или недопустимую структуру |

Валидатор проверяет структуру, полноту и целостность файлов. Он не может доказать личность участников или истинность человеческого наблюдения. Поэтому P5 переводится в ✅ только после фактической сессии и организационного утверждения акта; один успешный код `0` не меняет план автоматически.
