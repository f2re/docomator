# 🎨 Техническое задание на интерфейс Docomator

Версия: **1.0-draft**  
Статус: **нормативный UX/UI-контракт**  
Связанные документы: [REQUIREMENTS.md](REQUIREMENTS.md), [TECHNICAL_SPECIFICATION.md](TECHNICAL_SPECIFICATION.md), [ARCHITECTURE.md](ARCHITECTURE.md)

## 1. Цель

Интерфейс Docomator должен позволять человеку без подготовки понимать:

1. где он находится;
2. что система делает сейчас;
3. почему требуется ожидание или действие;
4. какие данные уже получены;
5. чего не хватает;
6. что произойдёт после нажатия кнопки;
7. как исправить ошибку;
8. сохранены ли изменения;
9. можно ли безопасно закрыть страницу;
10. где найти результат и историю операции.

> [!IMPORTANT]
> Пользователь не должен догадываться, «что происходит». Любая операция обязана иметь понятное название, текущий этап, результат и следующий безопасный шаг.

## 2. Визуальное направление

Интерфейс вдохновляется принципами macOS и iOS: ясной иерархией, системной типографикой, спокойными поверхностями, крупными зонами нажатия, предсказуемой навигацией и сдержанной анимацией. Это не означает копирование товарных знаков, системных иконок или закрытых компонентов Apple.

Основные свойства:

- минималистичная композиция без декоративного шума;
- одна основная задача и одна основная кнопка на экране;
- карточки с мягкими границами и умеренными тенями;
- системный стек шрифтов без внешней загрузки;
- светлая, тёмная и системная тема;
- спокойный акцентный цвет, не используемый как единственный носитель смысла;
- пояснения под нетривиальными полями серым вторичным текстом с достаточным контрастом;
- эмодзи допустимы как дружелюбные маркеры разделов и состояний, но не заменяют подпись и `aria-label`;
- анимация объясняет переход, но не задерживает работу.

## 3. Информационная архитектура

Основная навигация:

| Раздел | Назначение | Главный вопрос пользователя |
|---|---|---|
| 🏠 Обзор | готовность, следующий шаг, важные статусы | «Что мне делать сейчас?» |
| 🗂️ База знаний | типы, свойства, сущности и значения | «Какие данные знает система?» |
| 📄 Шаблоны | загрузка, анализ, поля и версии | «Как подключить документ?» |
| ✨ Документы | ручное создание, проверка и скачивание | «Как получить готовый документ?» |
| ⏱️ Автоматизации | расписания, события и история запусков | «Что формируется автоматически?» |
| ✅ Проверка | задачи оператора и неоднозначности | «Где требуется моё решение?» |
| ⚙️ Настройки | интеграции, роли, политика хранения | «Как настроена система?» |

На desktop используется боковая навигация. На узком экране — нижняя навигация с безопасной областью устройства. Редко используемые административные разделы скрываются за «Ещё» либо настройками роли.

## 4. Обязательная модель состояний

Каждый экран, карточка, список и длительная операция должны поддерживать применимые состояния:

| Состояние | Что видит пользователь |
|---|---|
| `idle` | доступное действие и краткое объяснение результата |
| `loading` | конкретный этап: «Проверяем шаблон», а не безымянный spinner |
| `empty` | почему данных нет и одна понятная кнопка для начала |
| `success` | что сохранено/создано и где найти результат |
| `warning` | что работает с ограничениями и каков риск |
| `error` | человеческая причина, сохранность введённых данных, повтор/исправление и correlation ID |
| `degraded` | какой компонент недоступен, что продолжает работать и что заблокировано |
| `disabled` | почему действие недоступно и что требуется для разблокировки |
| `planned` | честная отметка, что функция ещё не реализована, без имитации работы |
| `waiting_user` | какой ответ нужен и зачем |
| `waiting_review` | кто должен проверить, срок и последствия ожидания |
| `retry_scheduled` | причина повтора, номер попытки и время следующей попытки |
| `cancelled` | кем/почему отменено и можно ли начать заново |

Запрещены:

- бесконечный spinner без текста;
- фиктивный процент выполнения;
- исчезновение кнопки без объяснения;
- молчаливый retry;
- общий текст «Что-то пошло не так» без следующего шага;
- визуальное состояние, передаваемое только цветом;
- успешное уведомление до подтверждения backend-а.

## 5. Длительные операции

Для анализа шаблона, LLM-вызова, рендера, preview, автоматизации и доставки показываются:

```text
Название операции
Текущий этап
Что уже завершено
Что происходит сейчас
Что будет следующим
Время начала и последнего обновления
Можно ли уйти со страницы
Можно ли безопасно отменить
Correlation ID
```

Если операция персистентна, пользователь получает текст: «Можно закрыть страницу — работа продолжится на сервере». Если она не персистентна, интерфейс обязан предупредить до закрытия.

Пример:

```text
📄 Готовим комплект документов
✓ Данные собраны
● Заполняем DOCX — обычно до 15 секунд
○ Проверим структуру файла
○ Создадим предварительный просмотр

Можно перейти в другой раздел. Мы уведомим о результате.
```

## 6. Уведомления

Иерархия уведомлений:

1. **inline-пояснение** рядом с полем или проблемой;
2. **status ribbon** для состояния текущего экрана;
3. **toast** для подтверждения фонового или краткого действия;
4. **notification center** для завершившихся фоновых операций;
5. **modal/dialog** только когда без решения пользователя продолжение невозможно.

Уведомление отвечает минимум на два вопроса: «что произошло?» и «что делать дальше?». Ошибка дополнительно сообщает, сохранены ли введённые данные.

## 7. Формы

Каждое нетривиальное поле содержит:

- понятную подпись;
- пример значения;
- серую подсказку «зачем это нужно»;
- единицу измерения, если применимо;
- признак обязательности;
- локальную ошибку рядом с полем;
- сохранение введённого значения при серверной ошибке.

Правила:

- внутренние ключи не должны показываться обычному пользователю без объяснения;
- сложные параметры раскрываются постепенно;
- destructive action отделяется от основной кнопки и требует подтверждения;
- значения по умолчанию безопасны и объяснимы;
- длинная форма разбивается на короткие смысловые шаги;
- после успешного сохранения показывается результат, а не только закрывается окно;
- черновики длительных форм должны сохраняться автоматически в будущих workflow-модулях.

## 8. Текст и тон

Текст должен быть:

- конкретным: «Не найден получатель письма», а не «Ошибка данных»;
- нейтральным: без обвинения пользователя;
- коротким в основном потоке;
- подробным в раскрываемой справке;
- единообразным для одинаковых состояний;
- русскоязычным по умолчанию, без необъяснённых англоязычных терминов.

Пример ошибки:

```text
Не удалось сохранить свойство
Ключ person height содержит пробел. Используйте person.height.
Введённые данные остались в форме.
Correlation ID: 7f4…
[Исправить] [Скопировать ID]
```

## 9. Адаптивность

Базовые диапазоны:

- 320–619 px: один столбец, нижняя навигация, полноширинные действия;
- 620–819 px: компактный tablet-layout;
- 820–1079 px: боковая навигация уменьшенной ширины;
- 1080 px и шире: полноценная рабочая область и двух-/трёхколоночные сетки.

Требования:

- минимальная зона касания — 44 × 44 CSS px;
- учёт `safe-area-inset-*`;
- отсутствие горизонтального скролла на 320 px;
- диалоги не выходят за viewport;
- таблица на мобильном превращается в карточки либо получает объяснимый горизонтальный скролл;
- критичная кнопка остаётся доступна без точного наведения мышью.

## 10. Доступность

Целевой уровень — WCAG 2.2 AA.

Обязательно:

- семантические заголовки и landmarks;
- keyboard navigation и видимый focus;
- `aria-live` для результата операций;
- `aria-busy` для загружаемых коллекций;
- подписи для emoji/icon-only controls;
- контраст вторичного серого текста не ниже согласованного AA-порога;
- поддержка `prefers-reduced-motion`;
- отсутствие информации только в цвете;
- корректный focus return после dialog/drawer;
- skip-link;
- увеличение текста до 200% без потери функций.

## 11. Офлайн и приватность

UI не должен загружать в runtime:

- CDN-скрипты;
- внешние шрифты;
- внешние изображения;
- аналитику;
- crash-reporting SaaS;
- удалённые feature flags.

Все ассеты включаются в release bundle. Интерфейс явно показывает, что работает в локальном контуре. При потере связи с API он не утверждает, что «Интернет отсутствует», а сообщает точнее: «Нет связи с локальным сервером».

## 12. Компоненты первой версии

- App Shell: desktop sidebar + mobile bottom navigation;
- Top Bar: название, описание, primary action;
- Status Ribbon;
- Toast Region;
- Help Drawer;
- Empty/Error/Planned State;
- Metric Card;
- Collection Card;
- Segmented Control;
- Guided Dialog Form;
- Skeleton;
- Status/Pill;
- Future Operation Timeline и Review Task Card.

Компоненты строятся на design tokens. Локальные значения цвета, radius, spacing и shadow вне token-системы требуют обоснования.

## 13. Критерии приёмки UX

| ID | Критерий |
|---|---|
| UX-AC-001 | Новый пользователь без инструкции создаёт тип «Человек», свойство «Рост» и объект за один сценарий. |
| UX-AC-002 | При каждом сетевом действии сразу виден этап, а после ответа — подтверждение либо исправимая ошибка. |
| UX-AC-003 | При серверной ошибке введённые данные остаются в форме, отображается correlation ID и действие восстановления. |
| UX-AC-004 | Пустой список объясняет причину и предлагает одну основную кнопку. |
| UX-AC-005 | Недоступная функция помечена как запланированная и не имитирует выполнение. |
| UX-AC-006 | Интерфейс работает на 320 px, desktop и при системной тёмной теме. |
| UX-AC-007 | Все основные действия доступны клавиатурой и имеют видимый focus. |
| UX-AC-008 | При `prefers-reduced-motion` нет значимой анимации. |
| UX-AC-009 | Runtime UI не делает запросов к внешним доменам. |
| UX-AC-010 | Проверяющий всегда может ответить: текущий этап, причина ожидания, следующий шаг и результат. |

## 14. Definition of Done для интерфейсной функции

Функция не считается готовой, пока не реализованы:

- loading, empty, success, validation error, server error и degraded state;
- тексты текущего и следующего шага;
- keyboard/focus behavior;
- mobile layout;
- offline impact;
- automated tests применимых состояний;
- обновление требований, ТЗ и roadmap;
- честная отметка ограничений.
