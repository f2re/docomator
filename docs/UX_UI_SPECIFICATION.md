# 🎨 Техническое задание на интерфейс Docomator

Версия: **1.3-draft**
Статус: **нормативный договор интерфейса**  
Последнее обновление: **2026-07-15**

Связанные документы: [требования](REQUIREMENTS.md), [основное ТЗ](TECHNICAL_SPECIFICATION.md), [архитектура](ARCHITECTURE.md), [пространства и аудитории](SPACES_AND_AUDIENCES.md).

## 1. Цель

Интерфейс Docomator должен позволять человеку без подготовки понимать:

1. где он находится;
2. какое пространство выбрано;
3. что система делает сейчас;
4. почему требуется ожидание или действие;
5. какие данные уже получены;
6. чего не хватает;
7. что произойдёт после нажатия кнопки;
8. как исправить ошибку;
9. сохранены ли изменения;
10. можно ли безопасно закрыть страницу;
11. где найти результат и историю операции.

> [!IMPORTANT]
> Пользователь не должен догадываться, «что происходит». Любая операция обязана иметь понятное русское название, текущий этап, результат и следующий безопасный шаг.

## 2. Визуальное направление

Интерфейс опирается на принципы ясности macOS и iOS: спокойную иерархию, системную типографику, крупные области нажатия, предсказуемую навигацию и сдержанную анимацию. Это не означает копирование товарных знаков или закрытых компонентов Apple.

Основные свойства:

- минималистичная композиция без декоративного шума;
- одна основная задача и одна главная кнопка на экране;
- карточки с мягкими границами и умеренными тенями;
- системный набор шрифтов без внешней загрузки;
- светлая, тёмная и системная темы;
- спокойный акцентный цвет, не используемый как единственный носитель смысла;
- пояснения под нетривиальными полями серым вторичным текстом достаточного контраста;
- эмодзи допустимы как маркеры разделов и состояний, но не заменяют подпись;
- анимация объясняет переход, но не задерживает работу.

## 3. Информационная архитектура

| Раздел | Назначение | Главный вопрос пользователя |
|---|---|---|
| Главная | готовность и следующий незавершённый шаг | «Что мне сделать, чтобы получить документы?» |
| Сотрудники | карточки людей, поля и группы | «Для кого будут созданы документы и хватает ли данных?» |
| Шаблоны | подключение DOCX/XLSX и связь с понятными полями | «Какой документ заполнять?» |
| Создать документы | единый ручной выпуск от выбора шаблона до запуска | «Как получить личные карточки или сводный документ?» |
| Результаты | готовые файлы, комплекты, ошибки и история | «Где скачать то, что уже создано?» |
| Расписания | повторяющиеся выпуски и доставка | «Что и когда формируется автоматически?» |
| Настройки | разделы данных, дополнительные типы, интеграции и хранение | «Как изменить расширенные правила системы?» |

На компьютере используется боковая навигация. На узком экране — нижняя навигация с учётом безопасной области устройства. Текущий организационный раздел постоянно показан компактным переключателем контекста, но не превращается в самостоятельный обязательный этап каждого выпуска.

Универсальные типы сущностей, свойства, снимки состава, технические привязки и машинные ключи не являются разделами первого уровня. Они остаются доменной моделью и доступны только там, где нужны для расширенной настройки или диагностики.

## 4. Основные пошаговые процессы

### 4.1. Добавить сотрудника и его поля

```text
Сотрудники
→ Добавить сотрудника
→ ФИО и при необходимости основные сведения
→ Добавить поле прямо в карточке
→ выбрать понятный тип значения
→ сохранить карточку
→ предложить импорт списка или создание документов
```

Тип «Человек», стабильные ключи типа и полей, область применения и чувствительность получают безопасные системные значения по умолчанию. Если пользователь добавляет нестандартное поле, интерфейс явно спрашивает: «Добавить поле всем сотрудникам?», но не просит придумать техническое имя.

### 4.2. Подключить шаблон личной карточки

```text
Шаблоны
→ выбрать DOCX или XLSX
→ проверить и сохранить исходник
→ отметить заполняемые места
→ выбрать для каждого места поле сотрудника по русскому названию
→ пробно заполнить данными одного сотрудника
→ просмотреть PDF
→ подтвердить готовность шаблона
```

Координаты OOXML, SHA-256, ключи полей и имена внутренних привязок не показываются в основном процессе. Диагностический блок раскрывается отдельно и доступен только для чтения.

### 4.3. Создать личные карточки всем сотрудникам

```text
Создать документы
→ выбрать шаблон личной карточки
→ выбрать всех, группу или отдельных сотрудников
→ выбрать «По одному документу на каждого»
→ увидеть точное число будущих файлов
→ проверить и заполнить недостающие данные
→ подтвердить состав и запустить
→ открыть результаты и скачать файлы или ZIP-комплект
```

Сервер продолжает фиксировать неизменяемый снимок состава и внутренний режим `one_per_member`, но пользователь видит только понятный итог: «27 сотрудников → 27 личных карточек». Выбор сотрудников не дублируется на нескольких экранах; переход из группы или шаблона открывает тот же процесс с уже заполненным контекстом.

Во всех трёх процессах постоянно показаны:

- текущий организационный раздел;
- текущий шаг и число оставшихся шагов;
- сохранены ли введённые данные;
- что требуется от пользователя сейчас;
- одна основная кнопка продолжения;
- ожидаемое число документов;
- следующий шаг и место будущего результата.

Организационный раздел и группа не должны визуально смешиваться: раздел является текущим контекстом, группа — сохранённым набором сотрудников внутри него.

## 5. Обязательная модель состояний

Каждый экран, карточка, список и длительная операция поддерживают применимые состояния:

| Состояние | Что видит пользователь |
|---|---|
| ожидание действия | доступное действие и краткое объяснение результата |
| загрузка | конкретный этап: «Проверяем шаблон», а не безымянный индикатор |
| пусто | почему данных нет и одна понятная кнопка для начала |
| успешно | что сохранено или создано и где найти результат |
| предупреждение | что работает с ограничениями и каков риск |
| ошибка | понятная причина, сохранность данных, повтор/исправление и идентификатор операции |
| ограниченная работа | какой компонент недоступен, что продолжает работать и что заблокировано |
| недоступно | почему действие заблокировано и что требуется для разблокировки |
| запланировано | честная отметка, что функция ещё не реализована |
| ожидание пользователя | какой ответ нужен и зачем |
| ожидание проверки | кто должен проверить, срок и последствия ожидания |
| повтор назначен | причина, номер попытки и время следующей попытки |
| отменено | кем или почему отменено и можно ли начать заново |

Запрещены:

- бесконечный индикатор без пояснения;
- вымышленный процент выполнения;
- исчезновение кнопки без причины;
- молчаливый повтор;
- текст «Что-то пошло не так» без следующего шага;
- состояние, передаваемое только цветом;
- сообщение об успехе до подтверждения сервера.

## 6. Длительные операции

Для анализа шаблона, вызова модели, формирования, предварительного просмотра, автоматизации и доставки показываются:

```text
Название операции
Текущий этап
Что уже завершено
Что происходит сейчас
Что будет следующим
Время начала и последнего обновления
Можно ли уйти со страницы
Можно ли безопасно отменить
Идентификатор операции
```

Если операция сохраняется на сервере, пользователь получает сообщение: «Можно закрыть страницу — работа продолжится на сервере». В противном случае интерфейс предупреждает до закрытия.

Пример:

```text
📄 Готовим комплект документов
✓ Данные собраны
● Заполняем DOCX — обычно до 15 секунд
○ Проверим структуру файла
○ Создадим предварительный просмотр

Можно перейти в другой раздел. Мы сообщим о результате.
```

## 7. Уведомления

Иерархия уведомлений:

1. пояснение рядом с полем или проблемой;
2. строка состояния текущего экрана;
3. краткое всплывающее подтверждение;
4. центр уведомлений для фоновых операций;
5. диалог только когда без решения пользователя продолжение невозможно.

Центр фоновых операций размещён в существующем разделе «Результаты», а не в отдельном пункте навигации. Он читает сохраняемые состояния предпросмотра, формирования и доставки выбранного раздела с локального сервера, поэтому после перезагрузки показывает ту же операцию. Идентификаторы доступны только в раскрываемых технических сведениях.

Уведомление отвечает минимум на два вопроса: «что произошло?» и «что делать дальше?». Ошибка дополнительно сообщает, сохранены ли введённые данные.

## 8. Формы

Каждое нетривиальное поле содержит:

- понятную подпись;
- пример значения;
- серую подсказку «зачем это нужно»;
- единицу измерения, если применимо;
- признак обязательности;
- локальную ошибку рядом с полем;
- сохранение введённого значения при ошибке сервера.

Правила:

- внутренние ключи, UUID, координаты и имена режимов отсутствуют в обычных формах и назначаются сервером;
- технические сведения показываются только для чтения после явного раскрытия административного или диагностического блока;
- сложные параметры раскрываются постепенно;
- обычная форма нового поля спрашивает только понятное название и тип значения; дополнительные правила раскрываются отдельно;
- опасное действие отделяется от основной кнопки и требует подтверждения;
- значения по умолчанию безопасны и объяснимы;
- длинная форма разбивается на короткие смысловые шаги;
- после успешного сохранения показывается результат;
- черновики длительных форм должны сохраняться автоматически.

## 9. Язык и терминология

Основной язык интерфейса и сообщений — русский.

На русском должны быть:

- названия разделов, кнопок, полей и состояний;
- состояния, уведомления и ошибки;
- пояснения, справка и действия восстановления;
- сообщения установки и первого запуска;
- отчёты проверки и совместимости документов.

Необязательные англицизмы запрещены. Вместо них используются, например:

| Не показывать пользователю | Показывать |
|---|---|
| `backend` | серверная часть |
| `renderer` | модуль формирования |
| `guided flow` | пошаговый процесс |
| `snapshot ID` | идентификатор снимка |
| `correlation ID` | идентификатор операции |
| `owner / manager / editor / viewer` | владелец / руководитель / редактор / наблюдатель |
| `spinner` | индикатор ожидания |
| `planned` | запланировано |

Форматы, названия стандартов и машинные ключи (`DOCX`, `XLSX`, `SQLite`, `SMTP`, `audience.members`) допускаются, когда они необходимы, но сопровождаются русским объяснением.

Автоматическая проверка терминологии входит в `npm run check`.

## 10. Адаптивность

Базовые диапазоны:

- 320–619 пикселей: один столбец, нижняя навигация, полноширинные действия;
- 620–819 пикселей: компактная компоновка планшета;
- 820–1079 пикселей: боковая навигация уменьшенной ширины;
- 1080 пикселей и шире: полноценная рабочая область и многоколоночные сетки.

Требования:

- минимальная зона касания — 44 × 44 пикселя;
- учёт безопасной области устройства;
- отсутствие горизонтальной прокрутки на 320 пикселях;
- диалоги не выходят за область экрана;
- таблица на телефоне превращается в карточки либо получает объяснимую горизонтальную прокрутку;
- критичная кнопка остаётся доступна без точного наведения мышью.

## 11. Доступность

Целевой уровень — WCAG 2.2 AA.

Обязательно:

- семантические заголовки и области страницы;
- управление клавиатурой и видимый фокус;
- голосовое объявление результата операций;
- признак занятости загружаемых коллекций;
- подписи для эмодзи и кнопок без текста;
- достаточный контраст вторичного серого текста;
- поддержка уменьшения анимации;
- отсутствие информации только в цвете;
- возврат фокуса после закрытия диалога или панели;
- ссылка перехода к основному содержимому;
- увеличение текста до 200% без потери функций.

Автоматический Playwright/axe-аудит охватывает только машинно-выявляемые нарушения на ключевых состояниях: семантику, доступные имена и роли, проверяемый инструментом контраст. Успешный результат не означает полного соответствия WCAG 2.2 AA и не заменяет ручную проверку клавиатуры, порядка и возврата фокуса, экранного диктора, увеличения текста и пользовательскую приёмку.

## 12. Автономность и приватность

Интерфейс не загружает во время работы:

- внешние сценарии;
- внешние шрифты;
- внешние изображения;
- аналитику;
- облачные службы сбора ошибок;
- удалённые переключатели функций.

Все файлы включаются в автономный комплект. При потере связи интерфейс сообщает точно: «Нет связи с локальным сервером», а не делает предположений о состоянии Интернета.

- оболочка приложения;
- верхняя панель и основное действие;
- строка состояния;
- область кратких уведомлений;
- панель помощи;
- пустое, ошибочное и запланированное состояния;
- карточки показателей и коллекций;
- переключатель вкладок;
- пошаговая форма в диалоге;
- каркас загрузки;
- метки состояний;

Компоненты строятся на системе общих переменных оформления. Локальные значения цвета, отступов, скруглений и теней вне этой системы требуют обоснования.

## 14. Критерии приёмки

| ID | Критерий |
|---|---|
| UX-AC-001 | Новый пользователь без инструкции добавляет первого сотрудника и нужные поля его карточки в одном сценарии; стандартный тип «Человек» система подготавливает сама. |
| UX-AC-002 | При каждом сетевом действии сразу виден этап, а после ответа — подтверждение либо исправимая ошибка. |
| UX-AC-003 | При ошибке сервера введённые данные остаются в форме, отображается идентификатор операции и действие восстановления. |
| UX-AC-004 | Пустой список объясняет причину и предлагает одну основную кнопку. |
| UX-AC-005 | Недоступная функция помечена как запланированная и не имитирует выполнение. |
| UX-AC-006 | Интерфейс работает на ширине 320 пикселей, на компьютере и при системной тёмной теме. |
| UX-AC-007 | Все основные действия доступны клавиатурой и имеют видимый фокус. |
| UX-AC-008 | При включённом уменьшении движения нет значимой анимации. |
| UX-AC-009 | Интерфейс не делает запросов к внешним доменам. |
| UX-AC-010 | Проверяющий всегда может назвать текущий этап, причину ожидания, следующий шаг и результат. |
| UX-AC-011 | Пользователь выбирает пространство, группу или отмеченных людей и заранее видит: один документ или несколько. |
| UX-AC-012 | Обычный пользовательский сценарий не требует понимания английских терминов. |
| UX-AC-013 | Новый пользователь без инструкции добавляет сотрудника, создаёт поле «Должность», вводит значение и сохраняет карточку в одном сценарии без машинных ключей. |
| UX-AC-014 | Новый пользователь без инструкции подключает поле шаблона к полю сотрудника, выбирая русские названия, а не вводя совместимые технические ключи. |
| UX-AC-015 | Новый пользователь без инструкции выбирает всех сотрудников, получает прогноз N личных карточек, исправляет пропуски, запускает выпуск и находит ZIP-комплект в результатах. |
| UX-AC-016 | Основной путь не требует посещать универсальную схему данных и не показывает UUID, ключи, координаты OOXML, `snapshot` или `one_per_member`. |
| UX-AC-017 | Переход из сотрудника, группы или шаблона открывает один и тот же процесс создания документов с предзаполненным контекстом. |

## 15. Готовность интерфейсной функции

Функция не считается готовой, пока не реализованы:

- загрузка, пустое состояние, успех, ошибка проверки, ошибка сервера и ограниченная работа;
- тексты текущего и следующего шага;
- управление клавиатурой и фокусом;
- мобильная компоновка;
- влияние на автономную поставку;
