# План упрощения интерфейса и выпуска личных карточек

Статус: **утверждён; UX-1—UX-4 реализованы, для UX-0/UX-5 готовы автономный release-bound Playwright/axe gate и закрытый по умолчанию сбор доказательств, но целевой прогон, ручная и пользовательская приёмка ещё не выполнены**

Дата: **2026-07-16**

Приоритет: **первый продуктовый этап до новых эксплуатационных функций**

## 1. Зафиксированное решение

Основной интерфейс Docomator строится вокруг результата пользователя, а не вокруг универсальной модели данных:

```text
Сотрудники → Шаблон → Создать документы → Результаты
```

Универсальные сущности, свойства, машинные ключи, снимки состава и технические привязки сохраняются внутри домена. В обычном интерфейсе они заменяются предметными названиями и действиями. Расширенная настройка произвольных типов остаётся доступной в «Настройках», но не блокирует выпуск кадровых документов.

Первый эталонный сценарий:

> Добавить или импортировать сотрудников, подключить шаблон личной карточки, выбрать всех сотрудников, исправить недостающие сведения и получить по одному файлу на человека.

Требования: DATA-001—014, SPACE-008—015, TPL-001—030, DOC-003, DOC-011—019, UX-001—036, AC-027—029.

## 2. Что признано дефектом текущего интерфейса

1. Пользователь обязан придумывать ключи `person`, `person.height`, ключи разделов и групп, хотя это идентификаторы хранилища.
2. Добавление человека и определение полей разделены между «Схемой данных» и «Пространствами».
3. Карточки сотрудника с просмотром и изменением его полей нет; значения вводятся в основном как исправление ошибки перед выпуском.
4. Поле шаблона связывается с данными через совместимый машинный ключ вместо выбора понятного поля сотрудника.
5. Выбор состава дублируется в разделе данных и в форме запуска.
6. Создание документов спрятано внизу длинной страницы «Шаблоны», а раздел «Документы» выглядит как будущая функция или только хранилище результатов.
7. Одновременно показываются устаревшие статусы «Скоро», технические этапы, координаты, UUID, ключи и повторяющиеся карточки состояния.
8. Текущий выбранный раздел данных в экранах сотрудников и шаблонов может расходиться.

Это несоответствие UX-001—004, UX-013—014, UX-023, UX-026, UX-032—036 и UX-AC-001.

## 3. Целевая информационная архитектура

Основная навигация:

| Раздел | Основное действие |
|---|---|
| Главная | Продолжить ближайший незавершённый шаг или создать документы |
| Сотрудники | Добавить сотрудника или импортировать список |
| Шаблоны | Подключить и проверить шаблон |
| Создать документы | Выпустить личные карточки или сводный документ |
| Результаты | Скачать, просмотреть ошибку или повторить выпуск |
| Расписания | Настроить повторный выпуск |
| Настройки | Разделы данных, дополнительные типы, интеграции и диагностика |

Текущий организационный раздел показывается компактным переключателем в верхней панели. Если раздел один, переключатель не требует действия. «Схема данных» и «Пространства» удаляются из навигации первого уровня.

Терминология обычного сценария:

| Внутреннее понятие | Пользовательский текст |
|---|---|
| entity / entity type | сотрудник, организация или запись |
| property definition | поле карточки |
| space | раздел данных; в одном разделе название не акцентируется |
| audience | кого включить |
| audience snapshot | зафиксированный состав; показывается только при необходимости объяснить неизменность |
| one_per_member | по одному документу на каждого |
| aggregate | один сводный документ |
| stable key / UUID | не показывается; «технические сведения» только для диагностики |

## 4. Целевые сценарии

### 4.1. Сотрудник и поля карточки

```text
Сотрудники
┌─────────────────────────────────────────────┐
│ Сотрудники                         Добавить │
│ [Поиск]                    Импортировать…    │
│                                             │
│ Иванов Иван   Инженер   Карточка заполнена  │
│ Петрова Анна  —         Не хватает 2 полей  │
└─────────────────────────────────────────────┘
```

Короткая панель «Новый сотрудник» содержит ФИО и часто используемые поля. Действие «Добавить поле» открывается внутри той же карточки и спрашивает только название, например «Должность», и понятный тип: текст, число, дата, да/нет. Сервер назначает стабильный ключ и безопасные значения остальных параметров.

После сохранения пользователь видит карточку и два следующих действия: «Добавить ещё» и «Создать документы». Произвольные типы и специальные ограничения полей находятся в расширенной настройке.

### 4.2. Подключение шаблона

```text
1 Документ → 2 Поля → 3 Проверка → 4 Готово

┌──────────────────────┬──────────────────────┐
│ Живой лист документа │ Поле в этом месте    │
│                      │ [Должность       ▾]  │
│  ФИО: ________       │                      │
│  Должность: ____ ●   │ Продолжить           │
└──────────────────────┴──────────────────────┘
```

Пользователь выбирает место в документе и связывает его с полем сотрудника по русскому названию. Если подходящего поля нет, оно создаётся здесь же после явного подтверждения. Техническая координата и ключ привязки скрыты. Проверка, обратное чтение, PDF и неизменяемая активация сохраняются как обязательные серверные операции.

### 4.3. Личные карточки на всех сотрудников

```text
Шаблон → Сотрудники → Проверка → Запуск → Результаты

Личная карточка сотрудника
Все сотрудники · 27 человек
По одному документу на каждого

Итог: 27 сотрудников → 27 DOCX → один ZIP-комплект
                                      [Продолжить]
```

На проверке показываются только отсутствующие значения, сгруппированные по сотрудникам. Исправление сохраняет данные в карточке сотрудника. Перед запуском система показывает шаблон, состав, число файлов и место результата. После запуска пользователь попадает в «Результаты», где видит готовые файлы, частичные ошибки и повтор только неуспешных строк.

## 5. Визуальное направление

Направление: спокойная локальная рабочая среда в духе macOS, где главным объектом является сам документ.

- Цвета: фон `#F5F5F7`, поверхность `#FFFFFF`, основной текст `#1D1D1F`, вторичный `#6E6E73`, акцент `#3B78F2`, тёмный фон `#111318`.
- Типографика: локальный системный стек `-apple-system`, `BlinkMacSystemFont`, `Segoe UI`, `Arial`; моноширинный текст используется только в раскрытой диагностике.
- Компоновка: одна основная область, одна доминирующая кнопка, короткие панели до пяти полей, постепенное раскрытие дополнительных параметров.
- Иконки: единый локальный монохромный SVG-набор; эмодзи не используются как основная навигация или декор каждой карточки.
- Движение: один короткий переход панели или шага 160—200 мс; при `prefers-reduced-motion` переход отключён.
- Запоминающийся элемент: «живой лист» — миниатюра реального документа, которая показывает выбранные поля и заполненность по мере прохождения процесса. Остальная оболочка остаётся сдержанной.

В светлой и тёмной теме сохраняются одинаковая иерархия, контраст WCAG 2.2 AA и видимый фокус. Интерфейс работает на ширине 320 пикселей без горизонтальной прокрутки.

## 6. План реализации

### UX-0. Честные статусы и карта пути

1. Убрать ложные метки «Скоро» у работающих документов и расписаний.
2. Зафиксировать один текущий раздел данных для сотрудников, шаблонов и выпусков.
3. Перестроить Главную вокруг трёх состояний: сотрудники готовы, шаблон готов, можно создавать документы.
4. Подготовить кликабельный прототип основных экранов и проверить сценарий без подсказок до изменения серверных контрактов.

Критерий выхода: проверяющий с Главной называет следующий шаг и находит начало личных карточек одним действием.

### UX-1. Сотрудник и его поля одним сценарием

1. Добавить серверное назначение стабильных ключей, не зависящих от изменяемого русского названия, с уникальным ограничением и идемпотентным повтором.
2. Добавить новую добавочную миграцию или операцию начальной настройки для стандартного типа «Человек»; применённые миграции не изменять.
3. Создать прикладную операцию сохранения сотрудника и значений его карточки одной короткой транзакцией.
4. Реализовать список, карточку, добавление и изменение сотрудника.
5. Встроить «Добавить поле» с явным подтверждением глобального поля согласно DATA-009.
6. Оставить произвольные типы и технические сведения в расширенных настройках.

Критерий выхода: AC-027 и UX-AC-001/013 проходят без инструкции.

### UX-2. Импорт списка без технических терминов

1. Переработать существующий прототип CSV/XLSX-импорта; не считать его новой реализацией с нуля.
2. Автоматически предлагать ФИО, табельный номер, должность, подразделение и почту по заголовкам колонок.
3. Просить выбрать понятную колонку для обновления существующих сотрудников, не показывая ключи сущности, свойства или группы.
4. Показать предварительный итог: новые, обновлённые, без изменений и ошибочные строки.
5. Добавить защиту от массового обнуления, модульные и интеграционные тесты.

Критерий выхода: рабочий список импортируется и образует группу без ввода машинных идентификаторов.

### UX-3. Пошаговое подключение шаблона

1. Разделить текущую длинную страницу на четыре последовательных экрана.
2. Использовать единый выбранный организационный раздел, не зависящий от временного элемента загрузки файла.
3. Заменить ввод ключа поля выбором поля сотрудника.
4. Встроить создание недостающего поля с подтверждением.
5. Сохранить обязательные проверки исходника, структуры, обратного чтения, PDF и активации.
6. Убрать координаты, ключи и SHA-256 в раскрываемую диагностику.
7. После перезагрузки продолжать с проверенной серверной копии без повторного выбора локального файла; сверять раздел, исходник, SHA-256 и черновик до показа структуры.

Критерий выхода: UX-AC-014 проходит на новом DOCX и XLSX.

### UX-4. Единый выпуск личных карточек

1. Перенести рабочий запуск из конца «Шаблонов» в самостоятельный раздел «Создать документы».
2. Использовать одну форму выбора всех, группы или отдельных сотрудников; переходы из группы и шаблона только предзаполняют её.
3. По умолчанию предлагать «По одному документу на каждого» для шаблона личной карточки.
4. Показать точный прогноз N файлов до фиксации состава.
5. Перенести исправление пропусков в понятные карточки сотрудников без ключей свойств.
6. После запуска открыть «Результаты» и явно показать ZIP, отдельные файлы и частичные ошибки.

Критерий выхода: AC-028—029 и UX-AC-015/017 проходят на 1, 10 и 100 сотрудниках.

### UX-5. Визуальная разгрузка и приёмка

1. Ввести обновлённые переменные оформления, системную типографику, локальный SVG-набор и «живой лист».
2. Удалить повторяющиеся декоративные карточки, плашки, технические подписи и лишние доминирующие кнопки.
3. Добавить сквозные браузерные сценарии: сотрудник + поле, импорт списка, шаблон + N личных карточек + ZIP.
4. Добавить визуальные эталоны для 320, 768 и 1440 пикселей, светлой и тёмной темы.
5. Автоматизировать проверку машинно-выявляемых нарушений доступности ключевых экранов без исключения правил контраста.
6. Вручную проверить полные клавиатурные маршруты, порядок и возврат фокуса, экранный диктор, 200% текста, зоны нажатия, сохранение формы и идентификаторы операций при ошибках.
7. Выполнить `npm run check:language` и полный `npm run check`.

Критерий выхода: два новых пользователя выполняют три эталонных задания без устной инструкции; только после этого пользовательский сценарий M4 получает статус ✅.

## 7. Что не меняется

- Автономная работа и отсутствие внешних ресурсов.
- Модульный монолит, SQLite и детерминированные серверные операции.
- Неизменяемость применённых миграций и активных версий шаблонов.
- Проверка OOXML как недоверенного ZIP/XML.
- Корреляционные идентификаторы, идемпотентность и аудит.
- Целостность организационных разделов и неизменяемый снимок состава.
- Проверка обязательных данных, частичный успех и повтор только ошибок.
- Проверка PDF перед активацией и контролируемая доставка.

Сам UX-R1 не меняет архитектурную границу: UI остаётся адаптером, а доменные идентификаторы и решения принадлежат прикладному и серверному слоям. Выявленный ранее конфликт IAM отдельно разрешён принятым [ADR-0006](adr/0006-shared-trusted-workspace-without-iam.md): приложение работает без учётных записей и ролей, а разделы являются общей организационной структурой, а не границей доступа.

## 8. Риски и контроль

| Риск | Контроль |
|---|---|
| Автоматический ключ меняется при переименовании | ключ создаётся один раз сервером и затем неизменяем |
| Дубликаты при повторной отправке формы | ключ защиты от повторов и уникальное ограничение |
| Упрощение скрывает важную проверку | проверка остаётся обязательным шагом с понятным итогом |
| Универсальные типы становятся недоступны | расширенная настройка сохраняет полный функционал |
| Старые шаблоны теряют привязки | существующие ключи и активные версии не изменяются; новая UI-связь использует их внутренне |
| Перестройка ломает текущий выпуск | вертикали включаются по одной с обратной совместимостью API и регрессионными тестами |

## 9. Зафиксированные решения

1. Эталонный предмет первого сценария — «Сотрудник»; произвольные типы сохраняются в расширенных настройках.
2. Основная навигация: «Главная / Сотрудники / Шаблоны / Создать документы / Результаты / Расписания / Настройки».
3. Машинные ключи не вводятся в обычном интерфейсе и назначаются сервером.
4. Выпуск личных карточек всем сотрудникам является первым сквозным критерием приёмки.
5. UX-0—UX-5 имеют приоритет над новыми функциями расширения.

Основные изменения UX-0—UX-4 выполнены после подтверждения плана 15 июля 2026 года; UX-1—UX-4 закрыты в коде, а UX-0 сохраняет открытую приёмку переключателя нескольких разделов. Мастер после перезагрузки продолжает с проверенного серверного исходника и не требует повторно выбирать локальный файл; черновик сверяется с разделом, исходником и SHA-256. Для UX-5 добавлен Playwright-контур с Chromium, адаптивными размерами, темами, клавиатурой, уменьшением движения, сквозными сценариями и обязательным CI-прогоном. Axe-аудит без исключений проверяет машинно-выявляемые нарушения ключевых экранов на 320 пикселях в светлой теме и 1440 пикселях в тёмной. Проверенный offline bundle теперь содержит отдельный, не устанавливаемый в production Playwright/axe gate с target-specific Chromium; полный прогон сверяет target OS, установленный пакет, идентичность работающего API и связывает JSON с commit, manifest, release metadata и браузером. [Закрытый по умолчанию протокол](UX_ACCEPTANCE_PROTOCOL.md) fail-closed проверяет эти связи, сохраняет content-addressed копии и заполняет только машинные свидетельства. Он также проверяет полноту и SHA-256 ручных, визуальных и пользовательских файлов, но не создаёт и не утверждает их. Фактический целевой прогон, ручная проверка доступности, пользовательская приёмка двумя новыми пользователями и утверждение визуальных эталонов на Linux остаются открытыми.

## 10. Миграция и откат

- `0021_employee_profiles.sql` добавляет стандартный тип сотрудника и журнал защиты повторного создания карточки.
- `0022_employee_update_idempotency.sql` добавляет журнал защиты повторного изменения карточки; миграция `0021` после применения не изменялась.
- Обновление требует обычного запуска `npm run migrate` до readiness. До применения обеих миграций API намеренно остаётся в ограниченном состоянии.
- Обе миграции добавочные и не удаляют пользовательские данные. Автоматического обратного SQL-перехода нет: откат версии выполняется штатным восстановлением предварительной резервной копии базы и предыдущего неизменяемого каталога приложения.
- Новые пользовательские ключи создаются внутри сервера; прежние явные ключи и активированные шаблоны остаются совместимыми.
