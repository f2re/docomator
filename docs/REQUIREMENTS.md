# Требования к Docomator

Версия: **1.3-draft**
Статус: **нормативный документ проекта**
Последнее обновление: **2026-07-12**

## 1. Назначение и область

Docomator — автономная система для подключения произвольных DOCX/XLSX-шаблонов, разрешения вариативных полей из расширяемой базы знаний, ручного и автоматического формирования документов, проверки, архивирования и доставки.

Ключевые слова **MUST**, **SHOULD**, **MAY** определяют обязательность. В русскоязычном тексте им соответствуют «должно», «следует», «может».

Связанные документы:

- [Общее техническое задание](TECHNICAL_SPECIFICATION.md);
- [UX/UI-требования](UX_UI_SPECIFICATION.md);
- [пространства и аудитории документов](SPACES_AND_AUDIENCES.md);
- [архитектура](ARCHITECTURE.md).

## 2. Базовые ограничения

| ID | Требование | Приоритет |
|---|---|---:|
| SYS-001 | Система должна работать в автономном контуре без доступа в Интернет после установки. | MUST |
| SYS-002 | Целевая ОС: Debian GNU/Linux или Astra Linux Special Edition 1.7, x86-64; иные платформы допускаются после отдельного тестирования. | MUST |
| SYS-003 | Основной runtime — зафиксированная LTS-версия Node.js, поставляемая внутри release bundle. | MUST |
| SYS-004 | LLM должна выполняться локально через `llama.cpp/llama-server` на CPU и быть заменяемым компонентом. | MUST |
| SYS-005 | Уже активированный шаблон должен заполняться через форму при недоступной LLM. | MUST |
| SYS-006 | Архитектура первой промышленной версии — модульный монолит с процессами API, worker и llama-server. | MUST |
| SYS-007 | Система не должна требовать Redis, RabbitMQ, Kafka, Kubernetes или внешнюю векторную СУБД. | MUST |
| SYS-008 | LLM не должна исполнять код, SQL, shell-команды или напрямую изменять файлы и внешние системы. | MUST |

| SPACE-005 | Идентификатор ресурса из другого пространства не должен раскрывать или разрешать этот ресурс; операция отклоняется до мутации. | MUST |
| SPACE-006 | Внутри пространства должны поддерживаться редактируемые именованные группы с устойчивым порядком участников. | MUST |
| SPACE-007 | Участник группы обязан принадлежать тому же пространству; межпространственное членство запрещено на уровне backend и БД. | MUST |
| SPACE-008 | Аудитория документа выбирается из всех активных сущностей пространства, именованной группы или явно отмеченных сущностей. | MUST |
| SPACE-009 | Перед запуском создаётся неизменяемый снимок аудитории с порядком, отображаемыми данными, критериями, автором, временем и correlation ID. | MUST |

| SPACE-011 | Режим `one_per_member` должен создавать отдельную исполнимую единицу для каждого участника. | MUST |
| SPACE-012 | Режим `aggregate` должен создавать одну исполнимую единицу с упорядоченной коллекцией `audience.members`. | MUST |
| SPACE-013 | Template Compiler должен поддержать вывод `audience.members` как таблицы, списка, repeat row/range или иного разрешённого повторяющегося блока. | MUST |
| SPACE-014 | Document job и automation rule должны фиксировать `space_id`, audience snapshot и фактически использованный target mode. | MUST |
| SPACE-015 | Изменение именованной группы не должно менять уже созданный audience snapshot. | MUST |

## 6. Шаблоны и Document IR

| ID | Требование | Приоритет |
|---|---|---:|
| TPL-001 | Система должна принимать DOCX и XLSX и проверять MIME, ZIP/OOXML-структуру, размеры и relationships. | MUST |
| TPL-002 | Для импортированного файла должен строиться Document IR с устойчивыми IDs абзацев, runs, таблиц, ячеек, листов и диапазонов. | MUST |
| TPL-003 | Вариативные места должны обнаруживаться сначала детерминированными правилами, затем ограниченным LLM-агентом. | MUST |
| TPL-004 | Редактор должен вручную выделить текст, ячейку, строку, диапазон или блок и создать поле. | MUST |
| TPL-005 | Для DOCX основной привязкой является `w:sdt` с техническим tag `aifield:<uuid>`. | MUST |
| TPL-006 | Для XLSX основная скалярная привязка — defined name; служебные данные размещаются в `_AI_META`. | MUST |
| TPL-007 | Система должна поддерживать шаблон с заранее поставленными markers, пустой бланк и заполненные примеры для comparison-based detection. | SHOULD |
| TPL-008 | Манифест должен описывать поля, JSON Schema, bindings, источники, форматтеры, условия, повторы, review policy и тестовые данные. | MUST |
| TPL-009 | Активация допускается только после тестового рендера, структурной проверки и preview. | MUST |
| TPL-010 | Активированная версия шаблона неизменяема; изменение создаёт новую версию. | MUST |
| TPL-011 | Версия должна иметь уровень совместимости `safe-scalar`, `structured`, `generated` или `complex-office`. | MUST |
| TPL-012 | Должны поддерживаться комплекты из нескольких файлов с общим контекстом. | MUST |
| TPL-013 | Система должна формировать compatibility report для макросов, подписей, OLE, ActiveX, внешних связей и сложных объектов. | MUST |

## 20. Исключения первой версии

В первую промышленную версию не входят обязательная OCR-обработка сканов, квалифицированная электронная подпись, исполнение VBA/ActiveX/OLE, произвольный пользовательский код, полноценный BPMN-движок и кластерное горизонтальное масштабирование.

## 21. Управление изменениями требований

- Изменение MUST-требования требует обновления этого файла и при архитектурном влиянии — нового ADR.
- Pull request должен ссылаться на ID реализуемых требований.
- Roadmap отражает статус, но не переопределяет требования.
- Исключение из требования фиксируется отдельным ADR с рисками и сроком пересмотра.
- UI-функция без состояний loading/empty/error/degraded и без следующего шага не может считаться завершённой.
- Именованная группа редактируема, но audience snapshot после создания неизменяем.

- Изменение пользовательского текста требует проверки русской терминологии; внутренние машинные ключи не должны просачиваться в обычный интерфейс.
