# Требования к Docomator

Версия: **1.5-draft**  
Статус: **нормативный документ проекта**  
Последнее обновление: **2026-07-12**

## 1. Назначение и область

Docomator — автономная система для подключения произвольных DOCX/XLSX-шаблонов, разрешения вариативных полей из расширяемой базы знаний, ручного и автоматического формирования документов, проверки, архивирования и доставки.

Ключевые слова **MUST**, **SHOULD**, **MAY** определяют обязательность. В русскоязычном тексте им соответствуют «должно», «следует», «может».

Связанные документы:

## 2. Базовые ограничения

| ID | Требование | Приоритет |
|---|---|---:|
| SYS-001 | Система должна работать в автономном контуре без доступа в Интернет после установки. | MUST |
| SYS-002 | Целевая ОС: Debian GNU/Linux или Astra Linux Special Edition 1.7, x86-64; иные платформы допускаются после отдельного тестирования. | MUST |
| SYS-003 | Основная среда выполнения — зафиксированная LTS-версия Node.js, поставляемая внутри автономного комплекта. | MUST |
| SYS-004 | Локальная модель должна выполняться через `llama.cpp/llama-server` на центральном процессоре и быть заменяемым компонентом. | MUST |
| SYS-005 | Уже активированный шаблон должен заполняться через форму при недоступной локальной модели. | MUST |
| SYS-006 | Архитектура первой промышленной версии — модульный монолит с процессами службы API, фонового обработчика и `llama-server`. | MUST |
| SYS-007 | Система не должна требовать Redis, RabbitMQ, Kafka, Kubernetes или внешнюю векторную СУБД. | MUST |
| SYS-008 | Локальная модель не должна исполнять код, SQL, команды оболочки или напрямую изменять файлы и внешние системы. | MUST |

| ID | Требование | Приоритет |
|---|---|---:|

| SPACE-003 | Существующие записи при миграции должны быть помещены в детерминированное пространство `default`. | MUST |

| SPACE-005 | Идентификатор ресурса из другого пространства не должен раскрывать или разрешать этот ресурс; операция отклоняется до чтения чувствительных данных или изменения. | MUST |
| SPACE-006 | Внутри пространства должны поддерживаться редактируемые именованные группы с устойчивым порядком участников. | MUST |
| SPACE-007 | Участник группы обязан принадлежать тому же пространству; межпространственное членство запрещено на уровне серверной части и БД. | MUST |
| SPACE-008 | Аудитория документа выбирается из всех активных сущностей пространства, именованной группы или явно отмеченных сущностей. | MUST |
| SPACE-009 | Перед запуском создаётся неизменяемый снимок аудитории с порядком, отображаемыми данными, критериями, автором, временем и идентификатором операции. | MUST |

| SPACE-011 | Режим `one_per_member` должен создавать отдельную исполнимую единицу для каждого участника. | MUST |
| SPACE-012 | Режим `aggregate` должен создавать одну исполнимую единицу с упорядоченной коллекцией `audience.members`. | MUST |
| SPACE-013 | Компилятор шаблонов должен поддержать вывод `audience.members` как таблицы, списка, повторяющейся строки/диапазона или иного разрешённого блока. | MUST |
| SPACE-014 | Документное задание и правило автоматизации должны фиксировать `space_id`, снимок аудитории и фактически использованный режим результата. | MUST |
| SPACE-015 | Изменение именованной группы не должно менять уже созданный снимок аудитории. | MUST |
| SPACE-016 | Исходник шаблона, его структурное представление, поля, привязки и версии должны принадлежать одному пространству; межпространственное использование запрещено. | MUST |

## 6. Шаблоны, безопасный приём и структурное представление

| ID | Требование | Приоритет |
|---|---|---:|
| TPL-001 | Система должна принимать DOCX и XLSX и проверять MIME, ZIP/OOXML-структуру, размеры и связи. | MUST |
| TPL-002 | Для принятого файла должно строиться структурное представление с устойчивыми идентификаторами абзацев, текстовых фрагментов, таблиц, ячеек, листов и диапазонов. | MUST |
| TPL-003 | Вариативные места должны обнаруживаться сначала детерминированными правилами, затем ограниченным агентом локальной модели. | MUST |
| TPL-004 | Редактор должен вручную выделить текст, ячейку, строку, диапазон или блок и создать поле. | MUST |
| TPL-005 | Для DOCX основной привязкой является `w:sdt` с технической меткой `aifield:<uuid>`. | MUST |
| TPL-006 | Для XLSX основная скалярная привязка — именованный диапазон; служебные данные размещаются в `_AI_META`. | MUST |
| TPL-007 | Система должна поддерживать шаблон с заранее поставленными маркерами, пустой бланк и заполненные примеры для сравнения. | SHOULD |
| TPL-008 | Манифест должен описывать поля, JSON Schema, привязки, источники, форматтеры, условия, повторы, политику проверки и тестовые данные. | MUST |
| TPL-009 | Активация допускается только после тестового формирования, структурной проверки и предварительного просмотра. | MUST |
| TPL-010 | Активированная версия шаблона неизменяема; изменение создаёт новую версию. | MUST |
| TPL-011 | Версия должна иметь уровень совместимости `safe-scalar`, `structured`, `generated` или `complex-office`. | MUST |
| TPL-012 | Должны поддерживаться комплекты из нескольких файлов с общим контекстом. | MUST |
| TPL-013 | Система должна формировать отчёт совместимости для макросов, подписей, OLE, ActiveX, внешних связей и сложных объектов. | MUST |
| TPL-014 | Первичная проверка не должна распаковывать файл в пользовательский каталог, активировать шаблон или выполнять его содержимое. | MUST |
| TPL-015 | Отчёт первичной проверки должен различать допустимый файл, допустимый файл с замечаниями и отклонённый файл и объяснять следующий шаг по-русски. | MUST |
| TPL-016 | Сохранение проверенного исходника должно выполняться только после отдельного явного подтверждения пользователя и повторной серверной проверки. | MUST |
| TPL-017 | Принятый исходник должен храниться неизменяемо по SHA-256, быть привязан к одному пространству и сохранять полный отчёт, автора, время и идентификатор операции. | MUST |
| TPL-018 | Повторная загрузка одинаковых байтов не должна создавать второй физический объект или дубликат карантинной записи в том же пространстве. | MUST |
| TPL-019 | Отклонённый файл, файл с изменившейся контрольной суммой либо файл из другого пространства нельзя использовать для создания привязки поля. | MUST |
| TPL-020 | Структурное представление не должно возвращать браузеру исходный XML как исполняемое содержимое. | MUST |
| TPL-021 | Идентификатор элемента должен воспроизводимо рассчитываться из SHA-256 исходника и структурной координаты; повторный анализ неизменяемого файла должен возвращать те же идентификаторы. | MUST |
| TPL-022 | Ответ структурного анализа должен иметь настраиваемое ограничение количества элементов, сохранять полные итоговые количества и явно сообщать о сокращённом показе. | MUST |
| TPL-023 | DOCX-анализ должен поддерживать основной текст, колонтитулы, сноски, абзацы, текстовые фрагменты и положение внутри таблицы; XLSX-анализ — листы, ячейки, значения, общие/встроенные строки и формулы. | MUST |

## 7. Поля, источники и форматирование

| ID | Требование | Приоритет |
|---|---|---:|
| FLD-001 | Поле шаблона должно быть отделено от глобального свойства и от физической привязки в документе. | MUST |

| FLD-003 | Каждое использованное значение должно иметь происхождение, доступное в интерфейсе и аудите. | MUST |
| FLD-004 | Должны поддерживаться декларативные операции `concat`, `coalesce`, `if`, сравнение, арифметика, поиск, отображение, фильтр, объединение и форматтеры. | MUST |
| FLD-005 | Запрещены `eval`, `Function`, динамический импорт пользовательского кода и исполнение кода из манифеста. | MUST |
| FLD-006 | Форматтер ФИО должен поддерживать падеж, порядок частей, полную/краткую форму и инициалы. | MUST |
| FLD-007 | Автоматически полученная грамматическая форма должна быть помечена как предложенная; подтверждённая форма может сохраняться в профиле. | SHOULD |
| FLD-008 | Форматтеры должны поддерживать дату, дату-время, число, валюту, процент, единицу, адрес, телефон и реквизиты организации. | MUST |
| FLD-009 | Должны поддерживаться скалярное значение, форматированный текст, список, таблица, повторяющаяся строка/диапазон и изображение в пределах уровня совместимости. | MUST |
| FLD-010 | Все обязательные поля и межполевые зависимости валидируются до формирования. | MUST |

## 8. Агенты локальной модели

| ID | Требование | Приоритет |
|---|---|---:|
| AI-001 | Агент определяется версионируемой системной подсказкой, входной/выходной JSON Schema, лимитами, временем ожидания и запасным сценарием. | MUST |
| AI-002 | Все логические агенты используют одну загруженную модель, если конфигурация явно не предусматривает иное. | MUST |
| AI-003 | Ответ агента должен проверяться JSON Schema и бизнес-валидатором. | MUST |
| AI-004 | После невалидного ответа допускается не более одного исправляющего вызова. | MUST |
| AI-005 | Агент выбора шаблона не имеет права создавать идентификатор шаблона; он выбирает только из переданных кандидатов. | MUST |
| AI-006 | Агент извлечения значений извлекает только сообщённые значения и не создаёт идентификаторы сущностей. | MUST |
| AI-007 | Агент привязки возвращает координаты структурного представления, которые серверная часть проверяет перед изменением OOXML. | MUST |
| AI-008 | Агент создания текста возвращает ограниченные структурированные текстовые блоки, а не HTML/OOXML. | MUST |
| AI-009 | Содержимое пользовательского документа рассматривается как данные и не может менять системную подсказку или набор разрешённых действий. | MUST |
| AI-010 | Для вызова агента сохраняются версии агента, модели, подсказки и схемы, контрольные суммы, длительность и результат проверки. | MUST |
| AI-011 | Журналы агента должны поддерживать маскирование и режим хранения только контрольных сумм для чувствительных данных. | MUST |

## 9. Формирование и проверка документов

| ID | Требование | Приоритет |
|---|---|---:|
| DOC-001 | Ручной процесс должен поддерживать свободный запрос и явный выбор шаблона. | MUST |
| DOC-002 | Неоднозначные сущности должны предъявляться пользователю для выбора. | MUST |
| DOC-003 | Все поля должны быть доступны через динамическую форму независимо от чата. | MUST |
| DOC-004 | Созданный моделью текст и итоговые значения показываются до подтверждения. | MUST |
| DOC-005 | Модуль формирования должен изменять только разрешённые привязки и сохранять остальные части OOXML. | MUST |
| DOC-006 | После формирования выполняются проверки ZIP/XML, привязок, незаполненных маркеров и обратное чтение ожидаемых значений. | MUST |
| DOC-007 | LibreOffice без интерфейса используется для предварительного просмотра и при необходимости пересчёта формул, но не как основной модуль формирования. | MUST |
| DOC-008 | Ошибка предварительного просмотра блокирует автоматическую доставку, если политика явно не разрешает исключение. | MUST |
| DOC-009 | Исходный результат и предварительный просмотр хранятся как отдельные неизменяемые объекты с SHA-256. | MUST |
| DOC-010 | Повторное формирование с изменёнными данными создаёт новую ревизию и не перезаписывает прежнюю. | MUST |
| DOC-011 | Перед формированием пользователь должен видеть пространство, источник аудитории, точный состав и ожидаемое число документов. | MUST |
| DOC-012 | Для сводного документа модуль формирования должен получать коллекцию `audience.members`, а не случайно выбранного основного человека. | MUST |
| DOC-013 | Для индивидуального режима каждая исполнимая единица должна иметь собственного основного участника и одного участника в локальном контексте аудитории. | MUST |
| DOC-014 | Изменение именованной группы после начала запуска не должно менять уже созданный снимок аудитории. | MUST |

## 20. Исключения первой версии

В первую промышленную версию не входят обязательная OCR-обработка сканов, квалифицированная электронная подпись, исполнение VBA/ActiveX/OLE, произвольный пользовательский код, полноценный BPMN-движок и кластерное горизонтальное масштабирование.

## 21. Управление изменениями требований

- Изменение MUST-требования требует обновления этого файла и при архитектурном влиянии — нового ADR.
- Запрос на слияние должен ссылаться на идентификаторы реализуемых требований.
- План развития отражает статус, но не переопределяет требования.
- Исключение из требования фиксируется отдельным ADR с рисками и сроком пересмотра.
- Функция интерфейса без состояний загрузки, пустого списка, ошибки, ограниченной работы и следующего шага не считается завершённой.
- Именованная группа редактируема, но снимок аудитории после создания неизменяем.
- Изменение пользовательского текста требует проверки русской терминологии; внутренние машинные ключи не должны просачиваться в обычный интерфейс.
